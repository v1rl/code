[G - Substring Game](https://atcoder.jp/contests/abc433/tasks/abc433_g)

SAM 是解决很多字符串难题（如多模式串匹配、子串统计、最长公共子串等）的神器。

相比于后缀数组 (SA) 和后缀树 (Suffix Tree)，SAM 的优势在于它是一个**在线算法**，可以在 $O(N)$ 的时间复杂度内构建，且空间复杂度也是线性的 $O(N)$（虽然常数比 SA 大，但结构更灵活）。

以下是专门为你准备的硬核讲解，涵盖**理论基础**、**核心结构**、**构造流程**及**代码模板**。

------

### 1. 核心概念与直观理解

后缀自动机是一个**有向无环图 (DAG)**。

- **节点 (State)**：每个节点代表原串 $S$ 的一个**子串集合**。
- **边 (Transition)**：表示在当前子串后追加一个字符。
- **性质**：从初始节点 (Root) 出发，沿着边走的任意路径，都对应 $S$ 的一个**子串**；反之，$S$ 的所有子串都能在 SAM 中找到对应的路径。

#### 关键理论：`endpos` 等价类

这是理解 SAM 的灵魂。

对于 $S$ 的任意子串 $t$，我们要关注它在 $S$ 中出现的所有结束位置的集合，记为 $\text{endpos}(t)$。

- **定义**：如果两个子串 $u$ 和 $v$ 的 $\text{endpos}$ 集合完全相同（即 $\text{endpos}(u) = \text{endpos}(v)$），那么它们属于同一个**等价类**。
- **SAM 的节点**：SAM 的每一个节点就代表这样一个**等价类**。

重要结论：

同一等价类中子串的长度是**连续的**。设该节点中最长子串长度为 `len`，最短为 `minlen`（即`len[link[i]] + 1`)，则该节点包含长度为 `[minlen, len]` 的所有子串，且较短的串是较长串的**后缀**。

------

### 2. SAM 的双重结构

SAM 实际上包含两个图结构，它们共享节点，但边的含义不同：

#### A. 转移图 (Directed Acyclic Word Graph)

- **结构**：普通的转移边 `next[u][c] = v`。
- **含义**：在状态 $u$ 代表的子串后面接上字符 $c$，转移到状态 $v$。
- **作用**：用于做子串匹配。

> 任意子串所处的等价类，都可以在转移图上从初始点沿字符边移动得到
>
> 等价类节点中的所有子串，即转移图上从初始点到该等价类节点的所有字符串路径

#### B. 后缀树 (Parent Tree / Suffix Links)

- **结构**：由 `link` 指针构成的树。
- **含义**：对于状态 $u$，`link[u]` 指向 $u$ 中最长子串的**最长后缀**，且该后缀**不属于** $u$ 的等价类（即 $\text{endpos}$ 变大了）。
- **数学关系**：`minlen(u) = len(link[u]) + 1`。
- **作用**：压缩了后缀树的信息。沿着 `link` 往上跳，相当于不断不断地切掉子串的前缀（保留后缀），直到跳到 Root。**Parent Tree 本质上就是 $S$ 反串的后缀树。**

> 在后缀树中，越往上走等价类中子串越短，endpos中结尾点的数量越大(occ的值越大)
>
> 每次添加字符后最新的cur节点在后缀树中一定是叶子节点，它的lst节点也一定是叶子节点
> 叶子节点到根节点路径上所有等价类的子串集，即为当前叶子节点中最长子串（即当前字符串）的所有后缀

### 3. 构造算法 (增量法)

假设我们要为字符串 $S$ 构造 SAM。算法是逐个加入字符的。

设当前已经处理了前缀 $T$，现在要加入字符 $c$。

设 last 为对应 $T$ 整个串的状态。

**步骤如下：**

1. **新建状态 `cur`**：

   - 创建一个新节点 `cur`。
   - `len[cur] = len[last] + 1`。
   - `endpos` 集合必然包含当前位置 $i$。

2. **更新转移 (处理后缀)**：

   - 从 `last` 开始，沿着 `link` 往上跳（遍历 $T$ 的所有后缀），记当前节点为 $p$。
   - 如果 $p$ 没有字符 $c$ 的出边：`next[p][c] = cur`。继续 `p = link[p]`。
   - 如果跳到了根节点都没找到有 $c$ 出边的节点，说明 $c$ 以前没出现过，则 `link[cur] = root`，退出。

   ![2](images\2.png)

3. **处理冲突 (Split/Clone)**：

   - 如果在步骤 2 中，发现某个 $p$ 已经有了 $c$ 的出边，指向状态 $q$。
   - **Case A (理想情况)**：`len[q] == len[p] + 1`。
     - 这说明 $q$ 代表的最长串就是 $p$ 代表的最长串接上 $c$。
     - 直接让 `link[cur] = q` 即可。

   ![3](images\3.png)

   - **Case B (需要拆分)**：`len[q] > len[p] + 1`。
     - 这说明 $q$ 包含了比 $p+c$ 更长的串（来自于以前的某个上下文），现在`q` 同时承担了“短串”和“长串”的 endpos 信息，不合法。所以我们需要把 $q$ 拆成两部分：短串（长度小于等于`len[p] + 1`的部分），长串（ 剩余的长度大于`len[p] + 1`的部分）
     - **克隆 (Clone)**：新建节点 `clone`。
       - 设置 `len[clone] = len[p] + 1`。
       - 对于转移图，复制 $q$ 的 `adj` 给 `clone`节点，并将后缀树上所有原来对`q`节点的指向改为对`clone`节点，从具体操作上说，即沿着 `link` 链继续从 $p$ 往上走，凡是指向 $q$ 的 $c$ 转移，都改为指向 `clone`。
       - 对于后缀树，设置`clone`节点的`link`为原来`q`节点的`link`，然后设置`cur`节点和`q`节点的`link`为`clone`节点

   ![4](images\4.png)

注意：`clone`节点和`q`节点所代表的`endpos`，从创建到结束都是**一样的**，即`clone`节点对于`q`节点来说，只是“结构上的分裂”，而不是“语义上的分裂”。

------

### 5. 时空复杂度

- **时间复杂度**：$O(|S| \cdot \Sigma)$。如果需要处理的字符过多，则使用 `map` ，复杂度为大常数的 $O(|S| \log \Sigma)$。虽然有循环，但均摊分析证明是线性的。
- **空间复杂度**：$O(|S| \cdot \Sigma)$。
- **节点数**：对于长度为 $N$ 的串，节点数最多为 $2N - 1$。
- **边数**：边数最多为 $3N - 4$。

### 6. SAM 的常见应用 (针对 ICPC)

作为 ICPC 选手，你不需要死记硬背构造过程，但必须熟练掌握如何**利用 Parent Tree 和 DAWG 性质解题**。

1. **不同子串个数**：
   - 每个节点 $u$ 代表的子串个数是 `len[u] - len[link[u]]`。
   - 对所有节点求和即可。
2. **任意子串出现次数**：
   - 利用 Parent Tree。
   - 构造时，给每个状态设一个计数器 `cnt`。如果是 `sam_extend` 创建的新状态（非 clone），`cnt = 1`；clone 的状态 `cnt = 0`。
   - 建完图后，按照 `len` 从大到小（拓扑序）将 `cnt` 累加到 `link` 父节点上：`cnt[link[u]] += cnt[u]`。
3. **最长公共子串 (LCS)**：
   - 对串 $S$ 建 SAM。
   - 让串 $T$ 在 SAM 上跑。如果当前状态有 $T[i]$ 的转移，这就走；如果没有，就沿着 `link` 往上跳，直到有转移或跳到 Root。
   - 维护一个当前匹配长度 `res`，随时更新最大值 `ans`。
4. **第 K 大子串**：
   - 在 DAG 上预处理每个节点往后能走出多少条路径。
   - 类似字典树的查找逻辑。
5. **两个字符串任意位置为起始的最长公共前/后缀（LCP）**

### 7.维护节点endpos中结束点的个数occ（子串在原串中的个数）

很多初学者只是机械地背诵“新建节点赋 1，克隆节点赋 0，最后在 Link 树上求和”，但作为冲刺金牌的选手，你需要从 **Endpos 集合** 和 **Parent Tree (Link Tree)** 的本质关系来彻底理解它。

理解 `occ` 的维护逻辑，本质上就是理解 **子串是如何在 SAM 中“存在”的**。

------

### 7.1 核心定义：什么是 `occ`？

在 SAM 中，每个节点 $u$ 代表一个等价类，包含若干个子串。这些子串的 结束位置集合 (endpos) 都是完全一样的。

$$occ[u] = |\text{endpos}(u)|$$

也就是说，occ[u] 代表状态 $u$ 中的任意一个子串在原字符串 $S$ 中出现的总次数。

------

### 7.2 数据的来源：实点 vs 虚点

为了计算这个次数，我们不能在构建过程中动态维护（除非你用 LCT，那个后面说），通常采用 **“标记+汇总”** 的延迟计算策略。

我们将 SAM 中的节点分为两类逻辑角色：

#### A. “实点” (Prefix States) —— 原始贡献者

当我们调用 `extend(c)` 时，不仅是加入了一个字符，更是加入了一个**新的前缀**。

- 设当前处理到第 $i$ 个字符，生成了新节点 `cur` (或称 `np`)。
- 这个 `cur` 代表的最长子串就是整个前缀 $S[1...i]$。
- **物理意义**：这个前缀在当前位置 $i$ 实实在在地“结束”了一次。
- **操作**：所以在 `extend` 中，`occ[cur] = 1`。

#### B. “虚点” (Cloned States) —— 结构辅助者

当我们在 `extend` 过程中遇到 `q` 需要拆分出 `clone` 时：

- `clone` 节点的诞生是为了解决 $\text{endpos}$ 的分歧，它只是复制了 `q` 的结构属性。
- `clone` 节点本身并不代表“当前这一步刚刚结束的那个新前缀”。
- **操作**：所以在 `extend` 中，`occ[clone] = 0`。注意，这里赋 0 不是说它出现 0 次，而是说它 **“不包含当前这一步新产生的前缀结尾贡献”**。它的次数将完全来自于它的 Parent Tree 子节点。

------

### 7.3 汇总逻辑：Parent Tree 的性质

构建完 SAM 后，我们得到了一个 DAG 和一棵 Parent Tree。计算 `occ` 的核心依赖于 **Parent Tree (Link 树)** 的性质。

性质：

如果 link[v] = u，说明 $u$ 代表的子串集合 是 $v$ 代表的子串集合的 后缀。

推论：

既然 $u$ 是 $v$ 的后缀，那么 凡是 $v$ 出现的地方，$u$ 必然也出现了。

反之，$u$ 的出现位置集合 $\text{endpos}(u)$，是所有以 $u$ 为 Parent 的节点 $v$ 的 $\text{endpos}(v)$ 的并集，再加上 $u$ 自己作为完整前缀出现的那些位置。

$$\text{endpos}(u) = \left( \bigcup_{v \in \text{children}(u)} \text{endpos}(v) \right) \cup \{ \text{作为前缀出现的当前位置} \}$$

对应到次数上：

$$occ[u] = \sum_{link[v]=u} cnt[v] + [u \text{ is a prefix state}]$$

在代码实现中，因为我们之前已经把“作为前缀出现”的 1 赋值给 `cur` 节点了，而 `clone` 节点是 0。所以我们只需要单纯地 **在 Parent Tree 上做子树求和** 即可。

### 7.4 详细代码逻辑分析

让我们回到代码，逐行剖析。

#### 第一阶段：打标记 (在 `extend` 中)

C++

```cpp
void extend(char c) {
    int cur = sz++; 
    // ... 
    
    // 逻辑：cur 代表当前刚刚插入完的整个前缀 $S[0...i]$
    // 这个位置 i 显然是一个合法的结束位置，贡献为 1。
    cnt[cur] = 1; 
    
    // ... 中间省略 ...
    
    if (需要克隆) {
        int clone = sz++;
        // ... 复制 next 和 link ...
        
        // 逻辑：clone 是被“分裂”出来的中间态。
        // 它不是当前这个刚刚产生的前缀 $S[0...i]$。
        // 它的出现次数完全由引用它的更长子串（它的子节点）决定，
        // 所以这里初始贡献标记为 0。
        cnt[clone] = 0; 
        
        // ...
    }
}
```

#### 第二阶段：向上汇总 (在 `build_info` 中)

这是必须在 **所有字符插入完毕后** 进行的一步。因为 Link 树的结构在构建过程中是不断变化的（`link` 指针会变），只有构建完成，树形结构才稳定。

我们利用 **拓扑序（按 `len` 长度从大到小）** 进行反向遍历。因为 Parent Tree 中，父亲的 `len` 一定小于儿子，所以按 `len` 降序也就是自底向上遍历树。

C++

```cpp
int mxlen = len[lst];
vector<int> bk(mxlen + 1);
for(int i = 1; i <= cnt; i ++) {
    bk[len[i]] ++;
}
for(int i = 1; i <= mxlen; i ++) {
    bk[i] += bk[i - 1];
}
vector<int> rank(cnt + 1);
for(int i = 1; i <= cnt; i ++) {
    rank[bk[len[i]] --] = i;
}

for(int i = cnt; i >= 1; i --) {
    int x = rank[i];
    int p = link[x];
    if(p != -1) {
        occ[p] += occ[x];
    }
}
```

### 7.5 进阶视角：为什么不能动态维护？(LCT)

你可能会遇到这种思考题：**能不能不最后跑一遍基数排序，而在 `extend` 的时候实时知道每个节点的 `cnt`？**

**答案是可以，但代价很大。**

在 `extend` 过程中，`link` 边不仅会新增，还会**断开并重连**（比如 `clone` 发生时，`st[q].link` 会从原来的父节点断开，指向新的 `clone`；`st[cur].link` 也会指向 `clone`）。

如果我们想实时维护 `cnt`，这就变成了一个 **动态树 (Dynamic Tree)** 问题：

1. 即时的 `cnt` 更新相当于给 Parent Tree 上的一条链（从新节点到根）上的所有节点权值 `+1`。
2. `clone` 操作涉及 Link 边的 Cut 和 Link。

如果你使用 **Link-Cut Tree (LCT)** 来维护 Parent Tree，你可以做到实时查询任意子串出现次数。

- 每次 `extend`：`Access(last)`, `MakeRoot(root)`, `Link` 等操作，然后链加 1。
- 复杂度：$O(N \log N)$。

**结论**：在绝大多数 ICPC 题目中（求所有子串出现次数、特定子串出现次数），**离线 $O(N)$ 的桶排序汇总** 吊打 LCT。除非题目强制要求在线询问（每加入一个字符就问你某个子串出现了几次），否则不要用 LCT。

### 总结

cnt 的维护逻辑就一句话：

前缀位置（Real Node）自带 1 点贡献，这 1 点贡献沿着 Parent Tree（后缀关系）逆流而上，最终汇聚成每个子串的总出现次数。
