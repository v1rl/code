这是一个非常好的切入点。对于ICPC金牌选手来说，Hall定理和Gale-Ryser定理是**组合数学与图论**的交汇点。

很多时候，当题目让你求“是否存在一种匹配方案”或者“构造一个满足行列和的矩阵”时，如果我们直接跑网络流（Max Flow），可能会因为建图太复杂或者状态太多而超时。这时候，这两个定理提供了**非构造性的判定条件**，能将复杂度从图论算法的 $O(E\sqrt{V})$ 甚至更高，降维到简单的代数运算 $O(N \log N)$ 或 $O(N)$。

### 一、 Hall定理 (Hall's Marriage Theorem)

这是二分图匹配中最基础、最重要的定理。它解决了**二分图完美匹配的存在性判定**问题。

#### 1. 定理描述

设二分图 $G = (X, Y; E)$，我们想知道**是否存在一个匹配，使得 $X$ 中的每个点都被匹配上**（即 $X$-完美匹配）。

**Hall定理指出：**

$X$ 存在完美匹配的**充要条件**是：对于 $X$ 的任意子集 $S \subseteq X$，其邻居集合 $N(S)$ 的大小都至少等于 $S$ 的大小。

$$\forall S \subseteq X, \quad |N(S)| \ge |S|$$

其中 $N(S) = \{ y \in Y \mid \exists x \in S, (x, y) \in E \}$。

#### 2. 直观理解（婚姻隐喻）

假设 $X$ 是 $N$ 个男生，$Y$ 是 $N$ 个女生。每个男生只喜欢特定的一些女生。

如果我们想让所有男生都能娶到自己喜欢的女生，那么**瓶颈**在哪里？

瓶颈在于：如果有一组 $k$ 个男生（需求侧 $S$），他们共同喜欢的女生总数（供给侧 $N(S)$）少于 $k$ 个，那么这 $k$ 个男生里注定有人打光棍。

Hall定理说：**只要不存在这种“供不应求”的局部瓶颈，整体就一定能匹配成功。**

#### 3. 竞赛中的应用 (Hall定理的推广)

在ICPC中，直接枚举所有子集 $S$ 是 $2^{|X|}$ 的复杂度，不现实。Hall定理通常有两种用法：

1. **证明贪心的正确性**：如果题目允许你按某种顺序（比如权值排序）选点，你可以用Hall定理证明只要前 $k$ 个点满足条件，整体就满足。

2. **Dilworth定理与最小路径覆盖**：很多DAG的最长反链问题本质上是Hall定理的变形。

3. **Hall定理的亏格形式 (Defect Form)**：

   如果不仅是判断存在性，而是求最大匹配数。

   $$\text{最大匹配数} = |X| - \max_{S \subseteq X} (|S| - |N(S)|)$$

   也就是说，最大匹配数 = 总人数 - 最严重的“供不应求”缺口。

### 二、 Gale-Ryser 定理 (0-1 矩阵存在性定理)

这正是 **ABC424 G** 题背后的核心数学原理。它是Hall定理在“度数约束”下的推广。

#### 1. 问题背景

给定两个非负整数序列：

- 行和序列：$R = (r_1, r_2, \dots, r_n)$
- 列和序列：$C = (c_1, c_2, \dots, c_m)$

问：是否存在一个 $n \times m$ 的 **0-1矩阵**，使得第 $i$ 行的和恰好为 $r_i$，第 $j$ 列的和恰好为 $c_j$？

*(注：这个问题等价于构造一个二分图，左边节点度数为 $R$，右边节点度数为 $C$)*

#### 2. 定理描述

存在这样的0-1矩阵的**充要条件**是：

1. **总和相等**：$\sum r_i = \sum c_j$（这是显然的，总1的个数要守恒）。

2. **控制条件 (Majorization)**：

   不妨令行和为供给侧，列和为需求侧

   我们将和序列 $C$ **从大到小排序**，即 $c_1 \ge c_2 \ge \dots \ge c_n$。

   对于任意的 $k \in [1, m]$，必须满足：

   $$\sum_{i=1}^n \min(r_i, k) \ge \sum_{j=1}^k c_j$$

#### 3. 深度剖析 (为什么是这个公式？)

让我们来拆解这个不等式，你会发现它非常直观：

- 左边 $\sum_{i=1}^m \min(r_i, k)$为**供给侧**：

  这是**所有行**能提供给这 $k$ 列的 1 的最大潜力供给。

  - 对于第 $i$ 行，它总共有 $r_i$ 个 1。
  - 但是，它在选定的这 $k$ 列里，**最多只能贡献 $k$ 个 1**（因为一行在每一列只能填一个格，不能重叠）。
  - 同时，它也不能贡献超过它自身总数 $r_i$ 的 1。
  - 所以，第 $i$ 行对这 $k$ 列的贡献上限是 $\min(r_i, k)$。

- 右边 $\sum_{j=1}^k c_j$为**需求侧**：

​	这是矩阵中**“需求最大”的前 $k$ 列**所需要的 1 的总数。我们要优先满足这些“大胃王”列。

- **结论**：

  如果对于**任意**前 $k$ 列，所有行能提供的最大供给量 $\ge$ 这 $k$ 列的需求量，那么矩阵就一定存在。

#### 4. 回看 ABC424 G

现在你再看 ABC424 G 的题解核心公式：

$$\sum_{j=1}^k B_{p_j} \le \sum_{i=1}^N \min(A_i, k)$$

这**完全就是 Gale-Ryser 定理的直接应用**！

- $B_{p_j}$ 是列和（歌曲需要的演员数）。
- $A_i$ 是行和（演员能跳舞的次数）。
- 题目把“构造矩阵”变成了“选子集”，所以你不需要对 $B$ 序列求全和，而是要在DP中动态维护被选中的 $B$ 的子集和。
- 因为 Gale-Ryser 要求 $B$ 必须降序排列才能做前缀判定，所以我们在写 DP 时**必须先对歌曲按 $B$ 降序排序**。

### 二、 思维复盘：从零推导的全过程

假设我们现在面对这道题，脑子一片空白，如何一步步推导出解法？

#### 第一阶段：去繁就简，寻找判定基准

- **原始问题**：在 $M$ 首歌里选子集，最大化 $\sum C_j$，且能满足人员分配。
- **简化问题**：假设我们**已经选定**了一组歌 $\{S_1, S_2, \dots, S_k\}$，如何判断这组歌是否合法？
- **尝试建模**：
  - 左边点集：$N$ 个偶像，源点 $S$ 连向偶像 $i$，容量 $A_i$。
  - 右边点集：$k$ 首歌，歌曲 $j$ 连向汇点 $T$，容量 $B_j$。
  - 中间边：所有偶像都可以跳所有歌，容量 1。
  - **判定条件**：最大流是否等于 $\sum B_j$？

#### 第二阶段：从“跑流”进化到“不等式”

- **瓶颈思考**：如果用 DP 选歌，不可能每次转移都跑一次最大流。我们需要一个$O(1)$或者$O(N)$的公式来判断流是否满载。

- **直觉/定理联想**：

  - 什么时候最难满足要求？当然是选的歌**需要的人特别多**的时候。

  - 如果我们选的这 $k$ 首歌，按所需人数 $B$ **从大到小排序**，记为 $b_1 \ge b_2 \ge \dots \ge b_k$。

  - **直观贪心**：对于需求最大的前 $x$ 首歌，它们一共需要 $\sum_{i=1}^x b_i$ 个人次。

  - **供给上限**：这 $x$ 首歌，每个偶像最多只能贡献 1 次（因为每首歌只能上一次），同时偶像自身受 $A_i$ 限制。所以 $N$ 个偶像对于这 $x$ 首歌，最多能提供 $\sum_{p=1}^N \min(A_p, x)$ 个人次。

  - **得出结论（Gale-Ryser定理的变体）**：

    一组歌合法的充要条件是：将这组歌按 $B$ 降序排列后，对于任意 $x \in [1, k]$，都满足：

    $$\sum_{i=1}^x B_i \le \sum_{p=1}^N \min(A_p, x)$$



#### 第三阶段：构建动态规划

- **发现序的必要性**：既然判定条件依赖于“从大到小排序”，那我们的 DP 顺序**必须**遵循这个排序。
- **预处理**：先将所有 $M$ 首歌按 $B$ 降序排序。
- **定义状态**：我们需要什么信息来做下一次判定？
  - 公式右边 $\sum \min(A_p, x)$ 只和“选了多少首歌”($x$) 有关，与具体选了哪首无关。
  - 公式左边 $\sum B_i$ 需要被记录。
  - **DP状态**：$dp[i][j][s]$
    - $i$: 考虑了前 $i$ 首（排序后的）歌。
    - $j$: 实际选了 $j$ 首歌。
    - $s$: 选出的 $j$ 首歌的 $B$ 之和。
    - $val$: 最大热度 $\sum C$。
- **转移方程**：
  - 不选第 $i$ 首：$dp[i][j][s] \leftarrow dp[i-1][j][s]$
  - 选第 $i$ 首：$dp[i][j][s] \leftarrow dp[i-1][j-1][s-B_i] + C_i$
  - **合法性检查（Gatekeeper）**：在转移时，必须满足 $s \le \sum_{p=1}^N \min(A_p, j)$。如果不满足，这个状态就是非法的，设为 $-\infty$。

### 三、 为什么你的“费用流”思路卡住了？

你的直觉非常敏锐，**容量约束 + 代价最大化** 确实是费用流的特征。你卡住的地方在于：**“选子集”这个动作在费用流里很难直接表达。**

#### 1. 你的思路断点分析

- **费用流模型通常解决**：“给定图，给定流量需求，求最小/最大费用”。
- **本题难点**：流量需求（选哪些歌）是不确定的，是我们要求的变量。
- **尝试修正**：如果强行用费用流，通常需要引入“收益边”。例如，汇点 $T$ 向每首歌 $j$ 连一条边，容量 $B_j$，费用 $-C_j$（最小费用最大流）。但这里有个严重问题：**要么满流（选这首歌），要么零流（不选）。** 标准费用流是连续的，它可能会为了省钱只给这首歌流 $B_j - 1$ 的流量，但这在题目里是不合法的（必须恰好 $B_j$ 人）。
- **本质冲突**：**“离散的 0/1 选择（要么满要么空）”** 与 **“流的连续性”** 之间的冲突。解决这种冲突通常需要“带下界的最小费用循环流”或者“最大权闭合子图”，但由于 $B_j$ 这一侧的度数约束是刚性的，很难构造简单的闭合子图模型。

#### 2. 如何回归正确路线？

当你发现：

1. 题目是关于**二分图匹配**或**网络流**背景。
2. 但是要求**选子集**或者**判定可行性**比较复杂。
3. 数据范围 $N, M \le 100$（也就是 $N^3$ 或 $N^4$ 级别）。

**关键修正动作**：

> 不要试图去“跑”这个流，而是去**描述**这个流存在的**数学条件**。

一旦你从“建图”转向“找霍尔定理/流的割性质”，你就会发现 $B$ 值的排序规律，从而自然地过渡到 DP。**这是网络流题目在高阶竞赛中的一种常见变形：不需要你写 Dinic，需要你懂 Max-Flow Min-Cut Theorem 背后的代数含义。**

### 四、 通用解题方法论（针对陌生难题）

为了避免“脑中一片空白”，建议按照以下四步法（**P.R.O.C.**）进行思考：

#### 1. Pattern Recognition (模式识别)

- **看约束**：是计数？最优化？还是判定？
  - 本题：最优化（最大权）。
- **看数据范围**：
  - $N \le 20$ $\to$ 状压 DP。
  - $N \le 100$ $\to$ $O(N^3)$ 也就是区间 DP、背包 DP、网络流、匹配。
  - $N \le 10^5$ $\to$ 贪心、线段树、二分。
- **看背景**：本题显然是资源分配（人分配给歌），典型的二分图匹配模型。

#### 2. Reduction & Simplification (降维与简化)

- **固定变量法**：如果我**固定**了要选哪些歌，怎么做？（这是最关键的一步，能把复杂问题退化为经典问题）。
- **极端情况法**：如果所有 $A_i$ 都一样？如果所有 $B_j$ 都一样？

#### 3. Optimization via Transformation (转化优化)

- 如果第2步得到的是一个“网络流判定”，立刻问自己：**有没有定理（Hall/Gale-Ryser）可以替代跑流？**
- 如果判定需要排序，那么**排序**就是 DP 的顺序。
- 如果约束是前缀和形式，那么**前缀和**或**计数**就是 DP 的状态维度。

#### 4. Construction (构造求解)

- 根据第3步提取的维度（“前 $i$ 大”，“选了 $j$ 个”，“总需求 $s$”），定义 DP 状态。
- 检查空间和时间复杂度是否通过。

### 五、 顶尖选手的“编码流”描述

既然你已经有了思路，如何像顶尖选手一样行云流水地写代码？重点在于**模块化思维**和**预判性编码**。

**顶尖选手的心理表征（Mental Simulation）：**

> "这题是排序后的背包 DP。我要先写个结构体存歌。为了防止下标搞混，我会用 1-based indexing。预处理那个 $\min(A, k)$ 的和，叫它 `Limit[k]`。DP 数组大小是 `[101][10101]`，开大一点没问题。滚动数组有点烦，直接开三维或者两维暴力倒序也行，$N=100$ 随便跑。边界条件 `dp[0][0]=0`，其他负无穷。OK，开写。"

**编码流程步骤：**

1. **Skeleton (骨架)**:
   - 写好 `using namespace std;`，定义常量 `INF`。
   - 定义 `struct Song { int b; long long c; }`。
   - 重载 `<` 用于排序（$B$ 降序）。
2. **Input & Preprocessing (核心数据准备)**:
   - 读入 $A$。读入 $B, C$ 存入 `songs`。
   - **Action**: `sort(songs)`。
   - **Precompute**: 算出 `Limit[k]` 数组。即 `Limit[k] = sum(min(A[i], k))`。这是为了让 DP 内部逻辑极简，不要在 DP 循环里写求和逻辑。
3. **DP Initialization (状态定义)**:
   - 定义 `dp[M+1][N*M+1]`。
   - 初始化为 `-INF` (注意 `long long` 的负无穷)。
   - `dp[0][0] = 0`。
4. **The Core Loop (核心逻辑)**:
   - 外层循环 `i` 从 0 到 $M-1$ (遍历物品)。
   - 内层循环 `j` 从 `i+1` 到 1 (遍历个数，倒序以复用数组/或者直接用新的一层)。
   - 最内层 `s` 从 `MaxSum` 到 `songs[i].b` (遍历容量)。
   - **Check**: `if (dp[j-1][s-b] == -INF) continue;`
   - **Constraint**: `if (s <= Limit[j])`。
   - **Update**: `dp[j][s] = max(...)`。
5. **Answer Extraction (收尾)**:
   - 遍历所有合法的 `dp` 状态找最大值。

**示例代码片段（模拟顶尖选手的清晰逻辑）：**

C++

```
// 预处理部分，将复杂逻辑剥离出主循环
vector<long long> limit(M + 1, 0);
for (int k = 1; k <= M; k++) {
    for (int x : A) limit[k] += min(x, k);
}

// DP部分，逻辑极其干净
// dp[j][s] : 选了 j 首歌，总人数需求 s 的最大热度
// 采用滚动数组思想（倒序 j 和 s）
vector<vector<long long>> dp(M + 1, vector<long long>(N * M + 1, -1e18));
dp[0][0] = 0;

for (auto& song : songs) { // 遍历每一首歌（物品）
    int b = song.b;
    long long c = song.c;
    
    // 倒序枚举选了多少首歌（背包的体积1）
    for (int j = M; j >= 1; j--) {
        // 倒序枚举总人数（背包的体积2）
        for (int s = N * M; s >= b; s--) {
            // 前置状态必须合法
            if (dp[j-1][s-b] < -1e17) continue;
            
            // 核心约束检查：当前总人数 s 是否超过了 选 j 首歌的供给上限
            if (s <= limit[j]) {
                dp[j][s] = max(dp[j][s], dp[j-1][s-b] + c);
            }
        }
    }
}
```
