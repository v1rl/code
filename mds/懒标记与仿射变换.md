这确实是本题最绕、也是最精彩的地方。我们可以把它想象成一个**“加密通信”**或者**“懒人记账”**的过程。

为了让你彻底理解，我们先丢掉复杂的公式，建立一个直观的模型。

### 1. 核心模型：真实值 vs. 存档值

想象你是一个图书管理员（程序），你需要维护很多张卡片（颜色），每张卡片上写着一个数字。

但是，经常会有上级领导（题目逻辑）下达命令：“把所有卡片上的数字 $v$ 都改成 $S - v$！”

如果有 10 万张卡片，你一张张去改写数字会累死（超时）。

于是你想了个懒办法：

- **卡片上的字（Raw）**：一旦写上去，就不再涂改。

- **黑板上的规则（mul, add）**：你在墙上挂了一块黑板，写着两个数字 `mul` 和 `add`。

- 真实值（Real）：任何时候，你想知道某张卡片现在的真实数值是多少，你就用卡片上的字套用公式：

  

  $$\text{真实值} = \text{卡片字} \times \text{mul} + \text{add}$$

这样，当领导说“所有数字都要变”时，你只需要改黑板上的 `mul` 和 `add`，不用动卡片，**一瞬间**就完成了几十万张卡片的更新。

------

### 2. 为什么要维护 `mul` 和 `add`？

我们需要支持的操作是：**所有数字 $v$ 变成 $Target - v$**。

我们来看看数学上发生了什么：

- 现在的真实值是：$OldReal = Raw \times mul_{old} + add_{old}$
- 新的真实值必须是：$NewReal = Target - OldReal$

我们把第一行代入第二行：



$$NewReal = Target - (Raw \times mul_{old} + add_{old})$$

$$NewReal = Target - Raw \times mul_{old} - add_{old}$$

我们把带有 $Raw$ 的项整理一下：

$$NewReal = Raw \times (\mathbf{-mul_{old}}) + (\mathbf{Target - add_{old}})$$

你看，这依然符合 $Raw \times A + B$ 的格式！

所以，我们只需要把黑板上的数字更新一下：

1. **新的乘数**：$mul_{new} = -mul_{old}$
2. **新的加数**：$add_{new} = Target - add_{old}$

**这就是为什么能在 $O(1)$ 时间内更新所有颜色的原理。**

------

### 3. 具体操作原理（sum, add, remove）

`sum` 很简单，它就是**所有有效卡片的“真实值”的总和**。因为题目最终要用它来算 $S_i$。

#### (1) 全局变换 (Apply Transform)

- **操作**：领导说“所有人变成 $Target - v$”。

- Sum 的变化：假设有 $k$ 张卡片，原来的总和是 $Sum_{old}$。

  新的总和就是 $\sum (Target - v) = k \times Target - \sum v = k \times Target - Sum_{old}$。

  - 所以：`sum = count * Target - sum`。

- **Mul/Add 的变化**：如上面推导，`mul` 变反号，`add` 变 $Target - add$。

#### (2) 新增颜色 (Add Color)

假设你要加入一张新卡片（颜色 $c$），它的真实值要是 $Val$（比如在你的思路里，初始是 0）。

但是卡片上该写什么字（Raw）呢？

必须满足：$Raw \times mul + add = Val$。

逆运算算一下：

$$Raw = (Val - add) \div mul$$

(注：因为 mul 要么是 1 要么是 -1，除法等于乘法)

$$Raw = (Val - add) \times mul$$

- **操作**：算好这个 $Raw$，写在卡片上，扔进盒子里。
- **Sum 的变化**：既然加入了一个真实值为 $Val$ 的东西，那 `sum = sum + Val`。

#### (3) 移除颜色 (Remove Color)

你要把卡片 $c$ 拿出来了。它上面的字是 $Raw_c$。

- **操作**：直接扔掉。

- Sum 的变化：我们需要从总和里减去它的真实值。

  也就是 sum = sum - (Raw_c * mul + add)。

------

### 4. 模拟一遍，让你彻底看清

假设我们只看奇数位置。

初始状态：sum = 0, mul = 1, add = 0, count = 0。

#### 步骤 1：处理 $i=1$，需要新增颜色 Red，初始值设为 0。

- 我们希望 Red 的真实值是 0。
- 黑板上 `mul=1, add=0`。
- Red 卡片上该写多少？ $Raw = (0 - 0) \times 1 = 0$。
- **状态**：Red卡片(0)。`sum = 0`。

#### 步骤 2：处理完 $i=1$，进行全局变换。

- 假设 $S_{-1} = 0$（Target）。我们要把所有值 $v$ 变成 $0 - v$。
- 更新黑板：
  - `mul` 变成 $-1$。
  - `add` 变成 $0 - 0 = 0$。
- **检查 Red 的真实值**：$Raw \times mul + add = 0 \times (-1) + 0 = 0$。没变，正确（因为 $0-0=0$）。

#### 步骤 3：处理 $i=3$，需要新增颜色 Blue，初始值设为 0。

- **注意！** 现在的黑板是 `mul = -1, add = 0`。

- 我们希望 Blue 的真实值是 0。

- Blue 卡片上该写多少？

  

  $$Raw = (0 - add) \times mul = (0 - 0) \times (-1) = 0$$

- **状态**：Red卡片(0), Blue卡片(0)。

  - Red 真实值：$0 \times (-1) + 0 = 0$。
  - Blue 真实值：$0 \times (-1) + 0 = 0$。

- `sum = 0`。

#### 步骤 4：处理完 $i=3$，假设 $S_1 = 10$，进行全局变换。

- 我们要把所有值 $v$ 变成 $10 - v$。
- 更新黑板：
  - `mul` 变成 $-(-1) = 1$。
  - `add` 变成 $10 - 0 = 10$。
  - `sum` 变成 $2 \times 10 - 0 = 20$（一共 2 个颜色）。
- **见证奇迹的时刻**：
  - **Red 的真实值**：$0 \times 1 + 10 = 10$。 （正确！之前是 0，变成了 $10-0$）
  - **Blue 的真实值**：$0 \times 1 + 10 = 10$。 （正确！之前是 0，变成了 $10-0$）

#### 步骤 5：处理 $i=5$，需要新增颜色 Green，初始值设为 0。

- 现在的黑板是 `mul = 1, add = 10`。

- 我们希望 Green 的真实值是 0。

- Green 卡片上该写多少？

  

  $$Raw = (0 - 10) \times 1 = -10$$

- **状态**：Red(0), Blue(0), Green(-10)。

- **验证 Green**：$-10 \times 1 + 10 = 0$。正确！

- `sum` 加上 0，还是 20。

#### 步骤 6：处理完 $i=5$，假设 $S_3 = 50$，进行全局变换。

- 所有值 $v \to 50 - v$。
- 更新黑板：
  - `mul` 变成 $-1$。
  - `add` 变成 $50 - 10 = 40$。
- **验证所有卡片**：
  - **Red (0)**: $0 \times (-1) + 40 = 40$。（原 10，变 $50-10=40$，对！）
  - **Blue (0)**: $0 \times (-1) + 40 = 40$。（原 10，变 $50-10=40$，对！）
  - **Green (-10)**: $-10 \times (-1) + 40 = 10 + 40 = 50$。（原 0，变 $50-0=50$，对！）

### 总结

1. 为什么这么维护？

   因为我们需要修改成千上万个数字，但每次修改的规则是统一的线性变换。通过维护变换规则（mul, add），我们可以不动原始数据，瞬间改变所有数据的含义。

2. Raw 值的作用：

   它是一个锚点。它记录了该颜色加入集合时的那一刻，相对于当时的变换规则的“相对位置”。

3. Sum 的变化：

   Sum 永远只在这个时刻统计所有卡片的当前真实值。变换时用数学公式批量更新 Sum，增删时单独加减具体的真实值。





> 这样的技巧在算法竞赛中有什么更常见，更被人熟知的定义和应用吗



这种技巧在算法竞赛（Competitive Programming）中非常经典，通常被称为 **“懒标记” (Lazy Tag / Lazy Propagation)** 的一种特例。

具体到本题这种同时维护“乘法”和“加法”的场景，它在数据结构题目中有一个非常著名的名字：**线段树 2 (Segment Tree with Affine Transformation)**。

以下是它在竞赛圈更通俗的定义、变体以及最经典的应用场景：

### 1. 最著名的定义：线段树的懒标记 (Lazy Tag)

你刚才在 `Tracker` 结构体里写的 `sum, mul, add`，其实就是一个**只有一个节点的线段树**。

在标准的**线段树**问题中，我们经常面临这样的操作：

- **操作 1**：将区间 $[L, R]$ 内的所有数字乘以 $k$。
- **操作 2**：将区间 $[L, R]$ 内的所有数字加上 $v$。
- **询问**：求区间 $[L, R]$ 的和。

**这与你的题目的区别在于**：

- 线段树处理的是**区间**操作（树形结构，$O(\log N)$）。
- 本题处理的是**全局**操作（相当于根节点，O(1)）。
- **核心逻辑完全一样**：维护 `mul` 和 `add` 两个标记，用来“偷懒”，不去真正更新底层的每个数字。

#### 经典题目：洛谷 P3373 【模板】线段树 2

这道题是所有算法竞赛选手的必修课。题目要求支持区间乘法、区间加法、区间求和。解法就是维护两个 tag：`mul` 和 `add`。

### 2. 数学定义：仿射变换 (Affine Transformation)

从数学角度看，这种形式 $f(x) = ax + b$ 被称为仿射变换。

它的核心优势在于封闭性（Composition）：两个仿射变换叠加之后，仍然是一个仿射变换。

- 变换 1：$y = k_1 x + b_1$
- 变换 2：$z = k_2 y + b_2$
- 叠加后：$z = k_2(k_1 x + b_1) + b_2 = (k_1 k_2) x + (k_2 b_1 + b_2)$

新的乘数是 $k_{new} = k_1 k_2$，新的加数是 $b_{new} = k_2 b_1 + b_2$。

这假定我们维护值为val，真实值为x就是为什么我们可以不断合并操作，而不需要重置数组的原因。

> 假定我们维护值为val，真实值为x
> 在算法中，通常是将形如 $x = kx + b$ 的重复变换用懒标记mul和add维护
> 初始时`val = x`，即`mul = 1，add = 0`
> 每次变换后`mul *= k，add = k * add + b`
> 任意次变换下，当前元素的值`val`总是满足`val = mul * x + add`

### 3. 其他常见应用场景

除了线段树，这种“全局标记”技巧还常用于以下场景：

#### A. 全局偏移量 (Global Offset)

这是最接近本题的应用。当题目只涉及**全局**的修改时，不需要线段树，直接维护全局变量。

- **场景**：虽然是一个数组，但每次操作都是“所有数加 x”或者“所有数乘 x”，或者“插入一个数”。
- **技巧**：
  - 维护全局 `global_add`。
  - **插入时**：存入 `val - global_add`（抵消当前的全局影响）。
  - **读取时**：读取 `stored_val + global_add`。
- **本题进阶版**：本题不仅有 `add` 还有 `mul`，所以插入时要进行逆运算 `(val - add) / mul`。

#### B. 启发式合并 (DSU on Tree / Small-to-Large Merging)

在树上问题中，有时我们需要合并两个集合（比如 `std::set` 或 `std::map`），并且给某个集合里的所有元素加上一个值。

- **暴力做法**：遍历集合每个元素加 $v$，复杂度 $O(Size)$。
- **优化做法**：给这个集合打一个 tag。如果要把集合 A 合并进集合 B，且 A 需要加值，我们可能只移动元素，并在移动过程中处理 tag 的差值。

#### C. 矩阵乘法 (Matrix Multiplication)

如果变换更复杂（比如斐波那契数列变换，或者多维向量变换），$ax+b$ 搞不定了，我们会把它升级为**矩阵乘法**。

- 维护一个全局矩阵 $M$。
- 原来的值向量 $\vec{v}$。
- 变换就是 $\vec{v'} = M \times \vec{v}$。
- 这本质上是仿射变换的高维推广。

### 4. 这里的坑点：优先级 (Order of Operations)

在维护 `mul` 和 `add` 时，算法竞赛中有一个经典的坑点：**先乘还是先加？**

通常约定：**先乘后加**。即 $val \times mul + add$。

为什么？因为这样在合并操作时更顺手：

- **当前状态**：$x \cdot mul + add$
- **操作：再加 $v$**：
  - $(x \cdot mul + add) + v = x \cdot mul + (add+v)$
  - 更新：`add += v`，`mul` 不变。
- **操作：再乘 $k$**：
  - $(x \cdot mul + add) \cdot k = x \cdot (mul \cdot k) + (add \cdot k)$
  - 更新：`mul *= k`，`add *= k`。

**注意**：如果是“先乘 $k$”，那么原来的加数 `add` 也要跟着乘 $k$。很多人在写线段树时会忘记给 `add` 乘 $k$，导致 Bug。这在本题的代码中也有体现：

C++

```
// 你的题目中：所有数变成 S - v  =>  v * (-1) + S
// 相当于：先乘 -1，再加 S
mul = -mul;      // 乘法标记累积
add = S - add;   // 这里的数学推导本质上就是 add_new = add_old * (-1) + S
```

### 总结

这个技巧在 CP 圈子里可以叫 “全局标记”，本质是 “懒标记 (Lazy Tag)” 的一种。

如果你和队友沟通，说 “维护一个全局的 tag，支持乘法和加法，插入的时候**逆运算处理**一下”，只要是做过线段树题目的队友都能秒懂。这是一个非常基础且强大的 $O(1)$ 优化手段。

> 线段树包括不断地对懒标记进行应用和初始化，所以插入时无需逆运算，而全局懒标记Tag不会初始化，执行插入则需要逆运算
