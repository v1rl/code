## 贪心



狭义的贪心：

每一步都做出在当前状态下最好或最优的选择，从而希望最终的结果是最好或最优的算法

> 不要去纠结严格证明，每个题都去追求严格证明，浪费时间、收益很低
>
> 大量累积贪心的经验，重点不是证明，而是题目的特征，以及贪心方式的特征，做好总结方便借鉴
>
> 关注题目数据量，题目的解可能来自贪心，也很可能不是，如果数据量允许，能不用贪心就不用（稳）
>
> 掌握用对数器验证的技巧

广义的贪心：

通过分析题目自身的特点和性质，只要发现让求解答案的过程得到加速的结论

> 最期望的情景是什么，以此为基础扩展到更普通普遍的情景
>
> 普通情景违背期望的原因是什么，原本符合期望的事物因为什么原因，被什么替代掉了
>
> 通过什么样的决策可以修正它

### 贪心是什么

**贪心算法**是一种**每一步都做当前看来最优选择**的方法。

> 一般来说都是寻找题目中的单调性

它的前提是：

当前短视的局部最优决策不会在后续被否定，即局部最优总是全局最优

它的核心想法是：

不回头、不枚举全局方案，而是在当前局部状态下，直接做一个最优决策，希望这些局部最优能拼成全局最优。
在每一步选择中都采取在当前状态下最好或最优的选择，从而希望导致结果是最好或最优的算法。它是一种极为短视的策略，不从整体最优上加以考虑，也不支持回溯，算法试图通过局部的目光短浅来拼凑出全局的宏伟蓝图。

所以它和暴力、搜索、动态规划最大的区别在于：

- **暴力 / 搜索**：把很多可能都试一遍
- **动态规划**：把所有子问题最优解系统合并，这个地方选或不选，都会记录下来，以后统一比较。它本质上允许“后悔”，因为所有可能都保存了
- **贪心**：每一步只选一个当前最优动作，直接往前走

你可以把它想成一种“**当场拍板**”的算法。

这种算法专门用来求解最优化问题。在很多核心算法中都能看到它的身影，比如图论中的单源最短路径和最小生成树，以及各种区间调度和资源分配问题。当问题需要求出某个最大值或者最小值，且数据规模庞大，使得动态规划或深度优先搜索的时间及空间复杂度完全无法承受时，通常就是贪心算法破局的时候。

### 贪心的实现

要实现贪心算法，通常有着非常标准的操作流程。首先是将实际问题抽象为数学模型。其次，也是最核心的一步，是寻找并确定一种排序规则或者优先级评估标准。最后，利用这个标准对数据进行处理，通过迭代一步步做出局部最优选择。在代码实现层面，这几乎总是伴随着对数组的自定义规则排序，或者使用优先队列等数据结构来动态维护当前的局部最优状态。

### 贪心的本质

要真正看透贪心的本质，脑海中不能只停留在每次拿最大的这种直观感受上。贪心之所以能在特定问题上严丝合缝地得到全局最优解，是因为这些问题严格具备两个核心性质：贪心选择性质和最优子结构。从更高的离散数学抽象层面来看，贪心算法实际上是在拟阵这种结构上寻找最大独立集的过程。只要问题的解空间结构能够被证明是一个拟阵，贪心策略就具有绝对的数学正确性。

### 贪心的证明

在面对复杂问题时判断能否使用贪心，需要依赖严密的逻辑验证，而不是直觉。最常用的验证武器是交换论证。你可以先假设存在一个任意的全局最优解序列，然后尝试交换其中两个相邻的元素，如果在你设定的贪心规则下，交换后的结果必然变差或者至少不变，这就从数学上证明了你的贪心策略是无懈可击的。



## 贪心与去除分支冗余——有限的操作次数

[1553. 吃掉 N 个橘子的最少天数 - 力扣（LeetCode）](https://leetcode.cn/problems/minimum-number-of-days-to-eat-n-oranges/description/)

这是一道非常经典的题目，你的 AC 代码写得极其优雅，尤其是利用 C++20 特性的 `auto dfs = [&](auto &&self, int x)` 实现了简洁的匿名递归，这已经是顶尖竞技选手的代码风格了。

针对你的目标——ICPC 金牌，仅仅“做对”确实不够，我们需要把这道题的价值榨干。以下是对这道题的全方位深度剖析。

### 1. 题目归类：这道题的本质是什么？

从表面上看，这道题的标签是“记忆化搜索”或“动态规划”。但站在 ICPC 的视角，更本质的归类应该是：**基于贪心策略的状态空间压缩**。

- **常规 DP/搜索**：处理的是连续、稠密的状态图，状态转移是步进式的（比如 $x \to x-1$），状态空间复杂度往往是 $O(N)$。
- **本题本质**：它是一个**稀疏状态图上的最短路问题**。由于 $N \le 2 \times 10^9$，连续状态图必定超时。但题目中 `/2` 和 `/3` 的操作具有极强的“收敛性”，通过贪心策略剔除掉无效的 `-1` 操作后，原本 $O(N)$ 的状态图被大幅压缩成了 $O(\log^2 N)$ 的稀疏图。

**核心考点**：如何识别并过滤掉冗余的转移边，将巨大的状态图坍缩成可计算的规模。

### 2. 思维复盘：假如没有上帝视角，该如何推导？

在赛场上遇到这道题，从零开始的推导链路应该是这样的：

**第一步：确立基准模型（Baseline）**

看完题目，最直观的反应是定义状态 $f(x)$ 表示吃完 $x$ 个橘子的最少天数。

转移方程呼之欲出：

$f(x) = \min(f(x-1), f(x/2) \text{ [如果偶数]}, f(x/3) \text{ [如果 3 的倍数]}) + 1$

复杂度：状态数 $O(N)$，对于 $N = 2 \times 10^9$ 绝对会 TLE 和 MLE。

**第二步：寻找瓶颈与突破口**

问自己：为什么状态数会这么多？

答：因为有 $x \to x-1$ 这个操作，导致状态是连续递减的。

再问自己：如果只有 `/2` 和 `/3`，状态数是多少？

答：极少。因为每次都是指数级缩小，最多只有 $\approx \log_2(N) \times \log_3(N)$ 种组合。

**第三步：性质挖掘（贪心视角的引入）**

既然 `-1` 操作是罪魁祸首，我们需要思考：**在什么情况下，我们真的需要执行 `-1` 操作？**

假设我现在有 1000 个橘子，我会选择连续吃 500 天每天 1 个吗？绝对不会，我肯定想尽快凑出 2 的倍数或 3 的倍数，然后使用除法。

**核心推论（贪心）**：连续的 `-1` 操作，**唯一**的目的就是为了让当前的橘子数变成 2 的倍数或 3 的倍数，从而可以执行 `/2` 或 `/3`。否则，连续使用 `-1` 永远劣于除法。

**第四步：重构状态转移（图的压缩）**

既然 `-1` 只是为了服务于除法，我们可以把“若干次 `-1` 再加一次除法”打包成一个**宏操作**：

- 为了使用 `/2`：我需要先减去 $x \% 2$ 个橘子，耗时 $x \% 2$ 天，然后再除以 2，耗时 1 天。总耗时：$x \% 2 + 1$ 天，后续状态变为 $\lfloor x/2 \rfloor$
- 为了使用 `/3`：我需要先减去 $x \% 3$ 个橘子，耗时 $x \% 3$ 天，然后再除以 3，耗时 1 天。总耗时：$x \% 3 + 1$ 天，后续状态变为 $\lfloor x/3 \rfloor$

此时，转移方程成功变异为：

$f(x) = \min(x \% 2 + 1 + f(\lfloor x/2 \rfloor), x \% 3 + 1 + f(\lfloor x/3 \rfloor))$

我们成功去除了纯粹的 $f(x-1)$ 转移分支，状态空间瞬间被压缩到了 $O(\log^2 N)$，题目迎刃而解。

### 3. 诊断你的初始思路

你的初始思路：“根据题目不难想到一种记忆化搜索的做法，但题目的数据范围很大，如果每次都按照 `/2`，`/3`，`-1` 三种策略去搜会超时，于是不会了。”

**你和正确思路差在哪？**

你卡在了**“对冗余分支的容忍度”**上。

当你发现 $O(N)$ 会超时时，你潜意识里认为“这三种策略是平权的、独立的”，既然全搜不行，那就说明整个框架都错了，从而陷入了“脑中一片空白”。

**下次该怎么思考才能回归正确思路线？**

遇到这种情况，牢记一个口诀：**“遇大看贪，遇巨找压”**（遇到大数据范围考虑贪心，遇到巨量状态考虑压缩）。

当你发现某个分支（比如本题的 `-1`）会导致状态爆炸，而其他分支（`/2` 和 `/3`）能让状态迅速收敛时，你的第一反应不应该是放弃，而是去**质疑那个爆炸的分支**。

问自己：“那个导致超时的分支，真的有必要在任意时刻都走吗？它能不能作为其他高效分支的辅助步骤被合并掉？”只要你开始质疑 `-1` 的独立存在的意义，你就能推导出上述的“打包操作”。

### 4. 通用解题方法论（应对陌生题目）

针对 ICPC 难度的算法题，遇到毫无头绪的题目，可以遵循以下四步剖析法：

1. **写出“暴力/朴素”数学模型**：不要一开始就想最优解。先用最直白的语言（比如普通的递归、暴力的多重循环）把逻辑理顺，确保你的第一版模型在逻辑上是绝对正确的。
2. **定位复杂度毒瘤**：盯着你的朴素模型，找出那个让复杂度飙升的变量或分支。是状态维度太多？是转移代价太大？还是某个操作导致了连续稠密的状态生成？
3. **寻找单调性、支配性或周期性**：
   - **支配性**：某个策略在大部分时候绝对优于另一个策略（本题中除法支配了减法）。
   - **单调性**：数据有没有明显的递增递减规律？
   - **周期性**：打个表，看看小数据范围内的答案是不是按某种规律循环。
4. **模型降维/等价转换**：利用第三步发现的性质，对第一步的模型进行手术。比如砍掉多余的图边（本题做法）、把一维状态升到二维换取转移的简便、或者利用数据结构（如线段树/单调队列）加速某一层循环。

### 5. 顶尖选手的编码流描述

脑中有了 $f(x) = \min(x \% 2 + 1 + f(x/2), x \% 3 + 1 + f(x/3))$ 这个式子后，如何不卡壳地写出来？顶尖选手的思维到手指的映射过程是这样的：

1. **确定数据流边界**：入参最大 $2 \times 10^9$，`int` 不会溢出，计算过程中的 `x%2 + 1` 等也不会溢出，全程使用 `int` 即可。返回值也是天数，`int` 足够。
2. **确立数据结构**：因为状态是跳跃的（比如直接从 $1000$ 跳到 $500$ 和 $333$），用数组做 DP 会爆内存且浪费，必须用哈希表做记忆化，`unordered_map<int, int>` 确立。
3. **搭建作用域（极其关键）**：为了避免将 map 定义成全局变量导致多组测试数据间的污染，直接在 `solve()` 函数内部定义 map，并使用 C++ 14/20 的 lambda 表达式写递归函数，闭包捕获外部的 map。
4. **填空式编写**：
   - 先写边界条件：`if (x <= 1) return x;`（0 需要 0 天，1 需要 1 天，完美合并）。
   - 再写记忆化拦截：`if (mp.count(x)) return mp[x];`
   - 最后写核心转移：直接 `return mp[x] = min(...)` 一气呵成。

你的代码已经完美契合了这个编码流，这说明你的代码工程素养非常好，缺少的仅仅是赛场上那一瞬间的数学思维转换。

### 6. 值得一提的扩展视角：常数级优化

虽然你的代码已经能 AC，但在 ICPC 赛场上，如果这题卡常数（比如多组测试数据 $T = 10^5$），`unordered_map` 的常数开销（哈希冲突、动态扩容）可能会成为致命弱点。

**进阶技巧：分段打表（或者叫阈值记忆化）**

由于递归到底层时，会出现大量小数据的重复查询。我们可以开一个大小为 $10^5$ 左右的普通数组（或者 `vector`）预处理出小范围的答案，或者在 DFS 时作为第一层缓存。

只有当 $x > 10^5$ 时，才去查 `unordered_map`；当 $x \le 10^5$ 时，直接 $O(1)$ 查数组。这在很多涉及大规模状态跳跃的题目中，能把耗时从几百毫秒硬生生压到几十毫秒。

## 单调性贪心——排序（指针扫描线）与堆 / **有序表**（频繁选择或淘汰最值）

[线段重合](https://www.nowcoder.com/practice/1ae8d0b6bb4e4bcdbf64ec491f63fc37)

> 左端排序有端堆

这道题是经典的区间处理问题，你的代码使用了“排序+优先队列（小根堆）”完美地达到了 $O(N \log N)$ 的时间复杂度，这正是这道题的最优解法之一。对于目标是 ICPC 金牌的选手来说，不仅要能 AC，更要能把题目看透。下面我们一步步来拆解。

### 1. 题目归类与本质剖析

这道题在算法竞赛中通常被归类为**区间问题（Interval Problems）**，其核心解法本质上属于**扫描线算法（Sweep-line Algorithm）\**结合\**贪心**的思想，并利用**堆（优先队列）进行动态集合维护**。

**它的本质是什么？**

这个问题的本质是**“动态维持有效状态”**。你可以把 X 轴想象成一条时间线，线段的左端点是“事件开始”，右端点是“事件结束”。题目求的“最多重合了几个线段”，本质上就是在问：**在这条时间轴上扫过去，同一时刻处于“活跃状态”（已经开始且尚未结束）的事件最多有几个。**

一旦把问题抽象成这样，你的视角就会从“一堆静态的线段”变成“一个随着时间推移不断变化的集合”。

### 2. 从零开始的思维推导复盘

如果不看题解，我们在赛场上该如何一步步推导出这个解法？

- **第一步：捕获关键条件与直觉降维**

  题目给了很多线段 $[start, end]$，求最大重合数。直觉上，如果我们在 X 轴上一个点一个点地去数，点是无限的，无法操作。但仔细一想，重合线段的数量**只有在遇到新的线段起点时才可能增加**。因此，X 轴上的连续空间被降维成了离散的关键点（即所有的 $start$ 和 $end$）。

- **第二步：确立扫描方向（单调性）**

  既然要在时间轴上统计，最自然的想法就是从左往右扫。为了实现“从左往右”，我们必须将杂乱无章的线段按照左端点 $start$ 进行升序排序。排序是解决一切区间问题的万灵药，它能人为制造出单调性，让我们按顺序处理问题。

- **第三步：寻找目标条件与状态转换（核心跃迁）**

  假设我们正在遍历排序后的线段。当我们扫到第 $i$ 个线段的左端点 $L$ 时，前面 $0$ 到 $i-1$ 个线段已经“开始”了。

  **自我问答：**前面这些线段，现在还有几个和我重合？

  **逻辑推断：\**由于已经按左端点排序，前面所有线段的起点必定 $\le L$。所以，决定它们是否与当前线段重合的\**唯一条件，就是它们的右端点 $R$ 是否 $> L$**。

- **第四步：数据结构选型（特异化修改模板）**

  现在的目标条件很清晰了：我们需要一个“容器”，把前面遍历过的线段的右端点装起来。每次遇到一个新的左端点 $L$，我们就要把容器里所有 $\le L$ 的右端点扔掉（因为它们已经不重合、过期了）。剩下的数量就是当前的重合数。

  **自我问答：**什么数据结构支持“不断加入新元素”且能“快速找出并删除最小的元素”？

  **逻辑推断：**小根堆（优先队列）。所以，我们将每个遍历到的区间的右端点扔进堆里，每次用堆顶和当前的 $L$ 比较，剔除过期数据。

### 3. 对你初始思路的诊断与纠偏

你最初的想法是：*“按照左端点排序，后面的所有节点左端点一定大于当前节点，但二分确定右侧的左端点小于当前节点右端点的点数量并不全。”*

**你的思路差在哪？**

你的思路卡在了**“静态的上帝视角”**。当你站在当前区间 $[L, R]$ 时，试图用二分法往后找有多少个区间的 $L' < R$。

这有两个致命的逻辑盲区：

1. **互相不重合：** 即使后面有 3 个区间的起点都小于你的 $R$，但这 3 个区间彼此之间可能早就一前一后错开了（完全不重合），你把它们加起来作为“最大重合数”是错误的。
2. **忽略了历史的留存：** 你只往后看，没有往前看。最大重合可能发生在你前面某个跨度极大的区间加上你，再加你后面的某个区间。二分法无法同时兼顾“之前尚未结束的”和“之后即将开始的”。

**下次如何回归正确思路线？**

在处理区间重叠、调度类问题时，**永远不要试图站在一个固定的区间去统计其他区间**。要把自己变成那根“从左向右扫的线”。不要问“当前区间和谁重合”，而是问**“当时间走到 $L$ 时，这个系统里还剩下谁”**。从静态的几何视角切换到动态的时间流视角，这是解决此类问题的破局点。

### 4. 遇挫时的通用解题方法论

对于志在 ICPC 的选手，遇到没见过的题脑子空白是常态。你可以建立以下这套“剥洋葱”思考框架：

1. **手玩样例与极端化画图（破冰）：** 不要空想。在纸上画几条线段，画极端情况（完全包含的、首尾相接的、错开的），拿一根笔竖着从左向右扫，观察笔扫过时重合数量的增减，你的身体直觉往往比大脑先找到规律。

4. **资源映射法（选型）：**
   - 需要频繁淘汰最小值/求最值 $\rightarrow$ 优先队列（堆）。

### 5. 顶尖选手的编码流描述

在脑中已经有了“排序+堆”的思路后，顶尖选手写代码时不是在“翻译逻辑”，而是在“走流水线”。这是一种状态机式的编码肌肉记忆：

1. **数据就位：** 读入所有 $N$ 个线段，直接用 `vector<array<int, 2>>` 存好。敲下 `sort(a.begin(), a.end());`。这一步是本能，不需要过脑子。
2. **核心部件组装：** 脑子里立刻反映出我们需要维护有效右端点，且要从小到大淘汰。直接盲打出小根堆的模板代码：`priority_queue<int, vector<int>, greater<int>> heap;`，并准备一个 `ans = 0` 记录历史最高水位。
3. **流水线运作（核心循环）：** 写出 `for(int i = 0; i < n; i++)`。在这个循环里，选手的大脑里只有三个严格有序的动作：
   - **动作一：清理垃圾。** 新客人（新的线段）来了，时间来到了 $L$。立刻检查堆顶（最早结束的线段），如果它 $\le L$，说明它已经失效，毫不犹豫地 `while` 循环 `pop` 掉。
   - **动作二：接纳新客。** 垃圾清理完毕，把当前新客的右端点 $R$ 塞进堆里 `push(r)`，代表它成为了活跃状态的一员。
   - **动作三：拍照留念。** 此时堆里的元素个数，就是当前时刻的准确重合数。更新最高纪录 `ans = max(ans, (int)heap.size());`。

按这种“清理 -> 插入 -> 结算”的三步走流水线写出来的代码，不仅一气呵成，而且几乎不会有 Bug。

### 6. 值得一提的进阶角度：极致优雅的“差分/事件排序”解法

既然目标是金牌，你必须掌握这道题的另一种更“扫描线”的本质解法——离散化差分。

你的堆解法是在**遇到左端点时，被动地去清理右端点**。其实，我们可以把左端点和右端点直接拆开来看，把它们都视为独立在时间轴上发生的“事件”：

- 遇到一个左端点，重合度 $+1$。
- 遇到一个右端点，重合度 $-1$。

**思路：**

将一条线段 $[L, R]$ 拆分成两个事件：一个时间点是 $L$，权重是 $1$；另一个时间点是 $R$，权重是 $-1$。把所有 $2N$ 个事件放进一个数组里，**按照时间点从小到大排序**。（如果时间点相同，由于题目说首尾相接不算重合，必须先处理右端点离开事件，即先 $-1$ 再 $+1$）。

排好序后，只需要用一个变量 `current_overlap` 顺着扫一遍数组，累加权重，过程中的最大值就是答案。

这种写法不需要优先队列，常数极小，代码极其简练，是纯正扫描线思想的体现。你可以尝试用这种思路自己写一遍代码，这对你理解“将空间转化为一维时间事件”会有极大的帮助。祝你在接下来的算法训练中一路长虹，早日摘金。

### 一、 题目归类与本质剖析

这道题的精准归类是：**贪心算法 + 扫描线思想（配合优先队列维护动态集合）**。在一些经典算法书中，这类问题通常被放在“区间调度（Interval Scheduling）”的进阶章节。

每一天就是一个“槽位”（不可再生资源），每一个会议就是一个“带有有效期的请求”。问题的核心在于：在任意一个时刻，面对一堆当前可用的请求，我们该如何挥霍手里的这唯一一个槽位，才能保证未来能装下最多的请求？

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上初见此题，如何一步步推导出优先队列？

**1. 捕获关键与直觉排雷**

- **直觉一：按结束时间排序贪心？** 经典的区间调度问题（求最多不重叠区间）是按结束时间排序。但这题不同，一天可以参加一个会议，会议可以“推迟”到它的有效期内的任意一天。单纯按结束时间排序，如果会议的开始时间很晚，会破坏贪心策略。
- **直觉二：按长度排序？** 优先参加短的？不对，长度为 1 的会议（[3,3]）和长度为 10 的会议（[1,10]），如果在第 3 天，显然应该先参加 [3,3]，因为 [1,10] 以后还能参加。

**2. 核心矛盾转换（视角翻转）**

既然从“会议”的角度去挑选“时间”很难受（因为一个会议有多个时间点可选，存在后效性），我们就**翻转视角，从“时间”的角度去挑选“会议”**。

- 让时间轴 $d$ 成为主变量，一天一天往后走。
- **目标条件：** 在第 $d$ 天，我只能从“已经开始且还没结束”的会议中挑一个。

**3. 局部极值推导（贪心策略的诞生）**

现在站在第 $d$ 天，手里有一堆合法的候选会议。我该选谁？

- 会议 A：第 $d+1$ 天过期。
- 会议 B：第 $d+10$ 天过期。
- **灵魂拷问：** 如果我今天不参加 A，明天 A 就死了；如果我今天不参加 B，明天 B 还能活蹦乱跳。为了最大化收益，我必须**优先抢救最快要死掉的会议**。
- **结论：** 总是选择结束时间最早的会议。

**4. 算法与数据结构映射**

为了实现上述策略，我们需要一个“黑盒”：

- 随着时间推进，不断有新会议加入（插入操作）。

- 随着时间推进，有些会议自然死亡了（删除过期操作）。

- 每天我都需要在这个黑盒里，拿出一个结束时间最小的会议（查询并提取最小值操作）。

  支持动态插入、查询最值、删除最值——**优先队列（小根堆）**是唯一的完美契合者。

### 三、 通用解题方法论：降维打击框架

在赛场上遇到多维变量（这题有开始时间、结束时间两个维度）无从下手时，可以套用这个**“降维-定界-贪心”**三步走方法论：

1. **寻找主轴（降维）：**

   当对象包含区间或多个时间点时，强行建立一条虚拟的“时间扫描线”。让时间单调递增，将静态的区间问题转化为动态的时刻问题。

2. **定义视界（定界）：**

   当扫描线停在时刻 $t$ 时，问自己：“此时此刻，我能看到哪些元素？”（在本题中，就是 $start \le t$ 的元素）。把这些可见元素统统扔进一个“待定池”中。

3. **寻找局部最优（贪心）：**

   在当前的“待定池”中，设计一个评价标准，选出最迫切或最优的那个。**（评判标准通常是：谁的限制最紧，谁最容易变得不合法，就先处理谁）。**

4. **清理历史垃圾：**

   在每次决策前，必须把“待定池”中因为时间推移已经变得不合法的元素剔除（即本题中的 $end < d$）。

只要遇到带“区间”、“时效”、“可用时间段”字眼的题目，立刻把这个框架拿出来套，90% 的题都能迎刃而解。

### 四、 顶尖选手的编码流（消除卡壳）

很多时候思路都有了，写代码却修修补补，主要是因为**状态转移的顺序**在脑子里是乱的。对于这类扫描线优先队列题，顶尖选手脑子里有一个极其标准的数据流水线架构：**In -> Clean -> Consume**。

你的代码已经体现了这个架构，这里我把它抽象成文字口诀，以后每次写这种题，按顺序敲这三个模块，绝对不会卡壳：

- **Phase 0: 数据预处理。**

  先把事件按开始时间排序（为了后续能按顺序进入流水线）。

- **Phase 1: 时钟驱动。**

  用一个循环代表时间流逝（`for d` 或 `while`）。

- **Phase 2: 进水管（In）。**

  把所有今天刚开始的事件（$start == d$）全部 push 进堆中。这叫**扩充侯选集**。

- **Phase 3: 排污管（Clean）。**

  检查堆顶，把所有已经过期的事件（$end < d$）全部 pop 掉。这叫**清理脏数据**。（注意：只看堆顶就行，不需要遍历堆，因为堆顶是最容易过期的，如果堆顶都没过期，下面的更不可能过期）。

- **Phase 4: 消费（Consume）。**

  经过上两步，现在堆里剩下的**全是合法且活着的**事件。直接取堆顶（最紧迫的），`pop` 掉，答案 `+1`。

只要你在敲代码时，脑子里默念 **“先压入新来的，再踢出过期的，最后吃掉最紧迫的”**，代码自然如行云流水，且 Bug 率极低。

你的 $O(D)$ 外层循环会直接 Time Limit Exceeded (TLE)。因为如果第一天有一个会议，第 $10^9$ 天有一个会议，中间全是真空期，你的代码会白白循环 10 亿次。

时间轴 $d$ 应该直接跳跃到**下一个会议的开始时间**，或者**当前手里正在开的会议的结束时间**。

### 一、 题目归类与本质剖析

这道题的算法类型可以精准归类为：**多路归并（K-way Merge）** 或 **带色彩的滑动窗口（Sliding Window with Colors）**。

**本质剖析：**

这道题的本质是**“多维度的短板效应”**。

我们手里有 $K$ 个变量（每个列表出一个数），这 $K$ 个变量的极差（Max - Min）决定了区间的长度。

要缩小极差，我们唯一的手段是：**要么降低最大值，要么提高最小值。** 所有的算法推导，都是围绕如何高效地执行这一“逼近”过程展开的。

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上，我们面临这道题，如何从零推导出最优解？

**1. 捕获关键条件与直觉建模**

- **直觉：** 题目要求从 $K$ 个数组中各取一个数。最暴力的想法是枚举所有的组合，但这显然是指数级的，可以直接毙掉。
- **条件利用：** 题目给了一个非常强烈的暗示——**“非递减顺序排列”**。这是破局的核心。有序意味着存在**单调性**，我们可以通过指针的单方向移动来代替盲目的枚举。

**2. 极限简化与状态转换（贪心推导）**

- **降维思考：** 假设我们已经从 $K$ 个数组中各挑了第一个元素（因为数组有序，这是每个数组能提供的最小值）。
- 此时，区间被这 $K$ 个元素中的**最小值**和**最大值**框定了。
- **核心拷问：** 我现在想让区间变小，该怎么办？
  - **能缩小最大值吗？** 不能，因为每个数组目前的指针都指在最左边，往前退就没元素了。
  - **能增大最小值吗？** 能！那个提供最小值的数组，它的下一个元素肯定比现在大（或相等）。把它的指针往右移一步，最小值就可能变大，区间极差就有可能缩小！
- **目标条件建立：** 我们的核心任务变成了：维护 $K$ 个元素的集合，**不断地找出这 $K$ 个元素中的最小值，将其从集合中踢出，并把该元素所在数组的下一个元素加进来**，同时在此过程中记录出现过的最小极差。

**3. 算法定位与特异化**

- 既然需要“频繁查找最小值、删除最小值、插入新值”，这完美契合了**优先队列（小根堆）**的特性。
- 由于我们在弹出一个元素时，还需要知道它属于哪个数组，以及它在数组中的索引位置，因此堆里存储的不能只是一个数字，而应该是一个结构体或 `tuple`：`(value, list_index, element_index)`。

*(注：你的做法走的是另一条推导路线：忽略原始数组的有序性，直接全局排序打平，变成了“求包含所有颜色的最短连续子数组”，这也是非常经典的滑动窗口思维。两种思路都是金牌选手必须掌握的。)*

### 三、 通用解题方法论（应对脑中空白）

在区域赛中遇到毫无头绪的题目，特别是涉及到多维变量或区间的题目，可以使用这套**“降维 - 锁定瓶颈 - 动态逼近”**的方法论：

1. **极度简化（假设法）：**

   把 $N=10^5$ 变成 $N=2$。如果只有 2 个列表，怎么求最小区间？很自然就是双指针，谁小谁往右走。

   将 $N=2$ 的结论尝试推广到 $N=K$：既然 2 个的时候是谁小谁移动，那 $K$ 个的时候，是不是也是“谁最小谁移动”？

2. **寻找单调性（破局点）：**

   问自己：“如果我改变某一个变量，答案一定会向着某个特定方向变化吗？”

   在这题中，因为数组有序，所以“右移指针 $\rightarrow$ 值变大 $\rightarrow$ 最小值被拔高”构成了严格的单调递推链条。**只要找到单调性，贪心或双指针/滑动窗口的正确性就有了保证。**

3. **确定评价指标的“瓶颈（Bottleneck）”：**

   一个复杂状态往往由几个边缘变量决定。这道题中，$K$ 个数在中间怎么分布完全不重要，**唯一决定答案的就是 Min 和 Max**。一旦锁定了瓶颈，你的数据结构就只需要针对这两个瓶颈进行维护（用堆维护 Min，用一个变量打擂台维护 Max）。

### 四、 顶尖选手的编码流（消除卡壳）

你的代码虽然 AC，但因为采用了全局排序和 `multiset`，时间复杂度达到了 $O(V \log V)$（$V$ 为所有元素的总数）。而使用小根堆（多路归并）可以将复杂度降到严格的 $O(V \log K)$，在常数和理论上都更优。

以下是使用小根堆写这道题时，顶尖选手的**思维切分和编码流**。不要把所有逻辑糅合在一起想，按照标准流水线写：

1. **准备件（State Definition）：**

   定义清晰的包裹。

   `struct Node { int val, r, c; };`（值，属于哪个列表，属于该列表的第几个）。

   自定义小根堆比较器：`val` 小的优先。

2. **初始化流水线（Bootstrapping）：**

   - 建堆：`priority_queue<Node, ...>`
   - 全局极值变量：`int mx = -inf;`
   - 把所有列表的第 0 个元素全部压入堆中。在压入的**同时**，维护更新全局 `mx`。

3. **核心驱动引擎（The Engine）：**

   写一个死循环 `while (true)`。在这个循环里，逻辑是一条直线的：

   - **第一步（结算）：** 弹出堆顶（当前的最小值）。此时，手握当前的 `min` 和全局的 `max`，去更新最终答案 `ans_l, ans_r`。
   - **第二步（步进）：** 看看刚才弹出的那个元素，它所在的列表还有没有下一个元素。
     - 如果没有了：说明某个列表已经被掏空，不可能再凑齐 $K$ 个元素了，**直接 `break` 结束整个算法**。
     - 如果有：拿到下一个元素的值，**先用它更新全局的 `mx`**，然后把它压入堆中。

4. **收尾：**

   返回 `ans_l, ans_r`。

**为什么这个编码流不会卡壳？**

因为它的状态转移是**原子化**的。你不需要同时去想“我现在堆里有几个元素”、“最大值是不是被弹出了”（注意：**最大值不需要维护，只需要一直记录历史碰到过的最大值即可**，因为最小值一直在单调递增，如果加入的新元素比历史最大值小，极差的上限依然被原来的最大值卡着；如果比历史最大值大，那极差的上限自然就变大了。这是一个非常美妙的性质）。

### 五、 值得一提的角度：时空视角的二象性

这道题其实展示了算法竞赛中非常迷人的一种**视角二象性**，你的解法和多路归并解法恰好对应了这两种视角：

- **视角 A（你的解法：空间降维/全局宏观）：**

  无视各个列表的界限，把所有点打平在一条空间数轴上，通过“收集齐 $K$ 种龙珠”的最短连续段来求解。这种视角的威力在于普适性，它不需要数组本身是有序的。只要给定一堆带标签的散点，它都能求出结果。代价是全局排序的 $O(V \log V)$。

- **视角 B（多路归并解法：时间前推/局部微观）：**

  尊重各个列表的界限，就像有 $K$ 个并排的赛道。我们不看全局，只盯着最慢的那个选手（最小值），不断鞭策他往前跑。这种视角的威力在于极致利用了局部的“有序”信息，把复杂度压低到了 $O(V \log K)$。

在未来的训练中，当你用某一种视角（比如打平空间）感到受挫，或者复杂度卡在常数过不去时，立刻尝试在脑中切换到另一种视角（保留多个赛道，用指针推进）。这种视角的无缝切换，是打通高难度数据结构题和贪心题的任督二脉。

### 一、 题目归类与本质剖析

这道题的算法类型可以精准归类为：**可反悔贪心（Regret Greedy）**，在竞赛圈有时也简称为**反悔堆**问题。

**本质剖析：**

在资源（这里是“时间轴”）有限的单向序列中，我们追求某种收益的最大化（这里是“课程数量”）。普通的贪心算法是“一锤子买卖”，做出的选择不可更改，这往往要求问题具备严格的无后效性。而“可反悔贪心”允许我们在向前推进的过程中，**用更优的代价替换掉过去所付出的高昂代价**。

这道题的本质可以抽象为：**在一个动态增长的背包中，如何通过不断踢出“最占空间”的物品，来保证装入的物品数量最多。**

### 二、 从零开始的思维复盘（打破上帝视角）

假设我们在赛场上第一次遇到这道题，如何一步步推导出反悔堆？

**1. 捕获关键条件与降维（消除无序性）**

- **输入：** 两个维度的变量——课程耗时 $len$，截止时间 $lastD$。
- **直觉：** 如果有一堆课要上，常识告诉我们应该先上快截止的课。为了验证这个直觉，我们假设只选两门课 A 和 B。如果 $lastD_A < lastD_B$，先上 A 的合法条件是 $len_A + len_B \le lastD_B$，先上 B 的合法条件是 $len_B + len_A \le lastD_A$。显然，前者更容易满足。
- **推论：** **按截止时间 $lastD$ 升序排序**。这一步将问题从“无序集合的选择”降维成了“时间轴上的线性决策”。

**2. 顺水推舟与遇到阻碍（朴素贪心尝试）**

- 沿着排序后的列表往后走，遇到一门课，只要当前时间 $res + len \le lastD$，那就直接学。
- **阻碍出现：** 遇到一门课，发现 $res + len > lastD$，学不了了。难道直接丢弃吗？

**3. 破局点：化劣势为优势（引入“反悔”机制）**

- 此时我们需要灵魂拷问：当前这门课学不了，是因为前面的时间被占满了。**之前占满时间的那些课，真的都值得学吗？**
- 假设当前这门课耗时极短（比如 $len = 1$），而我之前为了增加 1 门课的进度，学了一门耗时极长（比如 $len = 100$）的课。
- 如果我“穿越”回去，把那门 100 小时的课退掉，换成这门 1 小时的课。我的**总课程数量没有变少**（退一换一），但我却**凭空省出了 99 小时的时间**！这 99 小时可能会让我未来能多上两三门课。

**4. 目标条件的转换与算法定位**

- **目标条件：** 当遇到学不了的课时，寻找已学课程中耗时最长的课。如果最长的课比当前课还长，就执行“退一换一”。
- **算法定位：** 我们需要一个数据结构，支持“动态插入元素”和“快速查询/删除最大值”。这正是**大根堆（优先队列）**的完美应用场景。

### 三、 通用解题方法论（面对陌生题目的拆解）

在向金牌冲击的过程中，遇到“完全没思路”的题是常态。以下是一套行之有效的拆解思维流：

- **第一步：特征提取与极度简化（看透本质）**

  剥离题目的背景故事。问自己：这是图论、数学、DP、还是数据结构？本题的本质就是“带截止时间的变分量调度”。

- **第二步：结合数据范围反推算法（缩小候选域）**

  看 $N$ 的范围。如果 $N = 10^5$ 或更大，算法复杂度必须是 $O(N)$ 或 $O(N \log N)$。这就直接排除了绝大多数 $O(N^2)$ 的动态规划，将思路死死锁定在**贪心、二分答案、线段树/树状数组、或者堆/单调队列优化**上。

- **第三步：控制变量与构造极端反例（打补丁）**

- **第四步：状态的降维投影**

### 四、 顶尖选手的编码流描述

脑海中有了反悔堆的思路，如何像顶尖选手一样不卡壳地写出代码？核心在于**将思维转变为数据流水线（Data Pipeline），而不是复杂的逻辑控制流**。

你的代码已经非常接近理想状态，下面是更顺畅的“流式编码”心法：

1. **数据整形（Data Shaping）：**

   不要边读边处理。先把输入数据按照我们的需要规整好。使用 `vector<array<int, 2>>` 或 `vector<pair<int, int>>` 读入，并**干脆利落**地调用 `sort`。明确排序基准是 `lastD`。

2. **状态容器初始化（State Initialization）：**

   在处理循环之前，清晰地定义维护状态的容器。

   - `int res = 0;` （当前已消耗的时间流）
   - `priority_queue<int> heap;` （后悔药仓库，默认大根堆，只存耗时）

3. **核心流水线（Core Pipeline）：**

   使用 `for (auto [lastD, len] : a)` 让数据像流水一样经过我们的检测站。

   - **常规通道：** `if (res + len <= lastD)` -> 恭喜，可以直接消化。`res += len; heap.push(len);`
   - **反悔通道：** 不能直接消化，尝试吃后悔药。此时直接判定 `if (heap.size() && heap.top() > len)` -> 吐出最差的历史，吞下当前的，完成时间压缩。`res += len - heap.top(); heap.pop(); heap.push(len);`

4. **收尾：**

   流水线结束，留在堆里的就是存活下来的最优解集合。`cout << heap.size();`

**编码流优化提示：**

在反悔通道中，你可以将 `res -= heap.top(); res += len;` 合并为 `res += len - heap.top();`。这样在潜意识里你是在做一个**“时间增量计算”**（这里的增量是负数，代表时间变短了），这比两步操作更符合数学直觉，代码也会显得更紧凑。

### 五、 值得一提的角度：贪心的底气（交换论证）

在 ICPC 赛场上，写贪心最怕的是“伪贪心”导致无尽的 WA 且找不到 Bug，最后发现是思路错了。

高水平选手在敲下反悔贪心的代码前，脑海里会飞速闪过一个**交换论证（Exchange Argument）**的证明：

- 假设最优解和我们的解在第 $k$ 步产生了分歧。
- 我们的策略是：把耗时 $T_{long}$ 的课换成了耗时 $T_{short}$ 的课（$T_{short} < T_{long}$）。
- 这导致总时间 $res$ 变小。由于剩下的课程截止时间都是递增的，总时间的缩小**绝对不会导致**后续原本合法的课程变得不合法。
- 因此，我们的交换操作在维持课程数量不变的前提下，让剩余可用时间变得更充裕，这**严格不劣于**原来的方案。

### 一、 题目归类与本质剖析

这道题的算法类型可以精准归类为：**数学推导 / 贪心**（部分同学也会用动态规划去做第一版，但最终由于数据范围 $n \le 1000$ 必须优化为数学方法）。

**本质剖析：**

这道题的本质是**“离散型变量的均值不等式极值问题”**。 无论绳子有多长，将其切分求乘积，最终都可以抽象为寻找一个**“最优的基础乘法因子”**。就像搭积木一样，我们要找出用哪种规格的积木（长度为几的绳子段）来拼凑总长度，能让乘积的“爆炸力”最强。

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上，我们从来没见过这道题，该怎么一步步逼近正确答案？

**1. 捕获关键与极端微观化**

- **动作：** 题目要求分成 $m$ 段（$m > 1$），求乘积最大。
- **微观观察：** 先看极小的数。如果切出来长度为 1，对乘积毫无贡献（$x \times 1 = x$），反而浪费了总长度。所以**绝对不能切出长度为 1 的段**（除非总长度只有 2 或 3，被迫切出 1）。
- **基本单位定性：** 既然不能有 1，那基本单位只能是 2、3、4、5...。但如果是 4，可以直接等分为 $2 \times 2 = 4$，收益没变；如果是 5，可以切分为 $2 \times 3 = 6$，收益 $6 > 5$。
- **核心结论：** 任何 $\ge 4$ 的数，继续切分都能得到更大或相等的乘积。因此，**绳子最终一定全是由 2 和 3 这两个“原子单位”组成的。**

**2. 核心矛盾转换与对决**

- 既然只能切成 2 和 3，那面临一个选择：**多切 2 还是多切 3？**
- 找一个公倍数来对决：总长度为 6 时。
  - 全切成 2：$2 + 2 + 2 = 6$，乘积 $2 \times 2 \times 2 = 8$
  - 全切成 3：$3 + 3 = 6$，乘积 $3 \times 3 = 9$
- **目标条件建立：** 因为 $9 > 8$，所以 **3 是比 2 更优秀的原子因子**。我们的策略极其明确：**疯狂切 3，直到不能切为止。**

**3. 处理边界（特异化修改）**

- 按照“疯狂切 3”的策略，切到最后只会有三种剩余情况：余 0、余 1、余 2。
- 余 0 和余 2 直接乘上去即可（2 已经是原子单位，不需要再动）。
- **阻碍出现（余 1）：** 如果最后剩下一个 1，按照之前的结论，1 对乘积无贡献。此时怎么补救？
- **回溯修正：** 从之前切好的 3 里面退回一个，把 $3 + 1$ 组合成 $4$，然后再切成 $2 \times 2$。因为 $2 \times 2 = 4 > 3 \times 1 = 3$。
- 这就是你代码中 `lst == 1` 分支为什么要 `qmi(3, d - 1) * 4` 的底层推导过程。

### 三、 诊断你的初始思路：“除二递归贪心”差在哪？

你最初的想法是：*根据题目猜了一个除二递归的贪心，每次都把当前数均等拆成两个数相乘。*

**这个思路极其常见，它的直觉来源是正确的：** 两个数和一定时，它们越接近，乘积越大（比如周长一定的矩形，正方形面积最大）。

**但它为什么错了？** 因为你的思路默认了“每次只切两半”（相当于限制了局部 $m=2$），而题目允许切**任意多段**。

- **举个反例：** 假设 $n = 14$。
  - **除二递归流：** 14 拆成 $7 \times 7$。7 再拆成 $3 \times 4$。最终结果是 $(3 \times 4) \times (3 \times 4) = 144$。
  - **正确贪心流（全切 3）：** $3 \times 3 \times 3 \times 3 \times 2 = 162$。
  - $144 < 162$，除二递归被无情粉碎。

**下次如何自我纠偏？**

在 ACM 赛场上，凭借直觉猜贪心结论是拿金牌的必备技能，但**不加验证的贪心就是送 WA**。当你脑子里冒出“均分切两半”时，必须立刻在草稿纸上用一个小数据（比如 $n=7$ 或 $n=14$）手动模拟一下。只要发现一次拆出来的结果不如手动乱凑的结果大，立刻毙掉这个贪心思路。

### 一、 题目归类与本质剖析

这道题的算法类型可以归类为：**前缀/后缀极值维护（Prefix/Suffix Extremes）**，或者说是**破坏单调性的边界寻找**。

**本质剖析：**

如果跳出具体的数字，从宏观视角看，一个包含“最短无序子数组”的数组一定可以被划分为三段：

- **左段 $A$**：完全有序，且 $A$ 中的所有元素都**小于等于**后面所有元素。
- **中段 $B$**：无序段（这就是我们要找的）。
- **右段 $C$**：完全有序，且 $C$ 中的所有元素都**大于等于**前面所有元素。

这就引出了这道题最底层的数学本质：**我们要找的不是“哪里无序”，而是“哪里破坏了全局的极值约束”。**

左边界 $l$ 是第一个被右侧某个更小元素“背刺”的位置；右边界 $r$ 是最后一个被左侧某个更大元素“压制”的位置。

------

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上初见此题，毫无头绪，如何一步步推导到你的 $O(N)$ 解法？

**1. 建立基准线（上帝视角的暴力降维）**

- **直觉：** “排序后对比”是最直观的。复制一个数组排序，然后左右两端向中间逼近，找到第一个不同的位置，这就是左右边界。
- **评估：** 复杂度 $O(N \log N)$。在 ICPC 赛场上，如果数据范围是 $10^5$，这个解法能 AC，但如果数据范围是 $10^7$，就会 TLE。我们必须把 $O(N \log N)$ 降到 $O(N)$。

**2. 剥离冗余信息（关键条件转换）**

- **发问：** 为什么排序能找到边界？因为排序后的数组代表了“绝对正确的位置”。
- **转换：** 元素 `a[i]` 如果不在正确的位置上，意味着什么？
  - 意味着在 `a[i]` 的**左边**，存在一个比它**大**的数（把它往后挤了）。
  - 或者在 `a[i]` 的**右边**，存在一个比它**小**的数（把它往前挤了）。
- **目标条件确立：** 我们不需要知道数组完全排好序是什么样，我们只需要知道：**当前元素是否打破了“前面所有数的最大值”的压制，或者“后面所有数的最小值”的托举。**

**3. 状态压缩与算法定位**

- 为了判断 `a[i]` 左边有没有比它大的数，难道每次都要遍历左边所有数吗？不需要。左边对 `a[i]` 威胁最大的，就是左边的**最大值**。
- 同理，右边对 `a[i]` 威胁最大的，就是右边的**最小值**。
- **推断算法：** 只需要维护一个“历史最大值”从左向右扫描，记录最后一个打破规则的位置 `r`；再维护一个“历史最小值”从右向左扫描，记录最后一个打破规则的位置 `l`。这就是你最终写出的代码。

面对陌生题目，大脑空白往往是因为“想一口气吃成胖子”，试图直接在大脑中生成最优解。顶尖选手的思考路径通常是**“降维打击 + 逐步逼近”**：

2. **暴力打底（极度重要）：** 永远先想最暴力的解法（如全排列、$O(N^2)$ 遍历、甚至排序对拍）。**暴力解法中往往藏着最优解的基因。**
3. **寻找单调性与冗余（核心跨越）：**
   - 问自己：“我在暴力求解时，哪些信息被重复计算了？”
   - 这道题中，$O(N^2)$ 找左侧比自己大的数就是冗余，因为中间过程的历史极值没有被记录。

   - 如果需要“动态区间的和/极值” -> 线段树/树状数组。

1. **阶段一：防御性初始化（Defensive Setup）**

这份代码写得非常漂亮！用 `tuple` 或 `array` 封装状态，直接利用优先队列的默认大根堆特性，核心逻辑集中在寥寥几行，完全是区域赛选手的手笔。

这道《最大开支》表面上是蓝桥杯的题，但在 ICPC 体系中，它代表了一类极其重要且出现频率极高的数学模型。如果你在平时训练中要将这道题给队员们讲透，或者为了在金牌区不假思索地秒杀它的变种，我们需要把它彻底拆解。

------

### 一、 题目归类与本质剖析

这道题的精准归类是：**基于边际收益的贪心（Marginal Benefit Greedy）**，在更宏观的运筹学或金牌算法语境中，这被称为**离散凸优化问题（Discrete Convex Optimization）**。

**本质剖析：**

这道题的本质是**“资源的最优分配”**。你有固定数量的资源（$N$ 个人），面对多个独立的槽位（$M$ 个项目）。每个槽位随着资源的投入，其产生的总价值是非线性的。

贪心算法之所以在这里能成立，**唯一的数学底层支撑是因为它的“边际收益是严格递减的”**。这就像吃包子，第一个包子最解饿（收益最大），第二个次之。只要满足这个性质，我们就可以无脑每次挑“当前这一口收益最大”的去吃。

------

### 二、 从零开始的思维复盘（打破上帝视角）

如果在赛场上看到这道题，如何严谨地推导出优先队列，而不是靠“直觉瞎猜”？

**1. 捕获关键与陷入困境（上帝视角的陷阱）**

- **直觉：** 既然总钱数是人数乘以票价，那我是不是应该算算把 $X$ 个人放进项目 $i$ 的总开销，然后搞个背包 DP？
- **困境：** $N$ 和 $M$ 都不小，如果是完全背包，复杂度直接爆炸。而且，总花费公式是 $Cost(X) = X \times (KX + B)$。这是一个二次函数，不是线性的，普通的贪心（比如平均分）肯定不对。

**2. 关键转换：降维打击（化整为零）**

- 遇到这种“分配总量求极值”的题，强迫自己切换视角：**不要看总数，看增量**。

- 如果我们手里只有 1 个人，该给他安排哪个项目？当然是挑单人花费最大的。

- 如果现在再多加 1 个人进入项目 $i$，总花费会增加多少？

  设 $Cost(X) = K \cdot X^2 + B \cdot X$

  边际收益（增量） $\Delta Cost = Cost(X) - Cost(X-1)$

  代入化简得到： $\Delta Cost = K(2X + 1) + B$

**3. 核心证明与算法敲定（贪心的底气）**

- **这是最关键的一步：** 题目中有一个隐含条件，由于“人越多单价越便宜”，所以这里的 $K \le 0$。
- 既然 $K$ 是负数或零，那么随着 $X$ 的增大， $\Delta Cost = K(2X + 1) + B$ 是一条**单调递减**的直线！
- **结论：** 这意味着每个项目的“价值增量”就像一摞从大到小叠放的硬币。我们要取 $N$ 枚硬币，要想总价值最大，当然是每次比较所有硬币堆的最顶端，把最大的一枚拿走。
- **算法映射：** 需要频繁查询最大值、删除最大值、插入新值 $\rightarrow$ **大根堆（优先队列）**。你的代码中 `nv = num * k + x + k * (num + 1)` 其实就是完美化简后的边际收益递推式。

------

### 三、 通用解题方法论（应对脑中空白）

以后遇到陌生的资源分配问题（比如分配士兵攻打城池、分配服务器处理任务等），一旦脑子空白，立刻套用这套**“边际收益三步法”**：

1. **写出单点总收益函数 $F(X)$**：先明确把 $X$ 个资源全砸在一个目标上，总收益怎么算。
2. **求离散导数（差分） $\Delta F = F(X) - F(X-1)$**：在草稿纸上暴力推导出这个差分表达式。
3. **判断单调性（决定算法命运）**：
   - 如果 $\Delta F$ 是**单调递减**的（收益递减）：直接无脑使用 **优先队列（大根堆）贪心**。
   - 如果 $\Delta F$ 是**单调递增**的（投入越多，单价涨得越疯，比如 $K > 0$）：贪心失效！这说明你一旦开始投某个项目，就应该把它投满。此时直接转入 **动态规划（DP）** 或 **枚举**。

------

### 四、 顶尖选手的编码流（消除卡壳）

你的代码结构已经很好了，但如果要做到“写码如行云流水，一次 AC 不卡壳”，脑子里的逻辑切分需要非常干净。这类“优先队列模拟”的标准流水线如下：

1. **第一步：数学隔离（最容易写错的地方）。**

   千万不要在 `while` 循环里去现推公式。在写循环前，先在纸上写死推导关系。

   比如：`current_delta = k * (2 * X - 1) + b`。

   或者像你这样维护递推，但一定要明确堆里存的是什么：`{边际收益, k, b, 当前人数X}`。

2. **第二步：大根堆初始化（布阵）。**

   遍历所有项目，把每个人数 $X=1$ 时的边际收益算出来。**注意防毒：** 如果一开始算出来的边际收益 $\le 0$，直接扔掉，不要进堆（虽然你的代码在 `while` 里面防了，但如果在初始化时拦住，堆的开销会更小）。

3. **第三步：消费循环（核心流）。**

   脑子里默念三件事：**取最值 -> 结算 -> 生存判定**。

   - **取最值：** 弹出堆顶，只要堆顶 $\le 0$，直接 `break`（提前拿满最大收益，剩下的全是亏钱的）。
   - **结算：** 答案加上堆顶收益，人数 $N$ 减 1。
   - **生存判定：** 用推好的公式算出下一阶段（$X+1$）的边际收益，**无脑 Push 回去**（哪怕它是负数，下一次取最值时也会触发 `break`，这样可以让循环体保持极简，不需要额外的 `if`）。

### 五、 值得一提的角度：数据范围的降维打击（冲金核心）

这才是这道题真正通往 ICPC 金牌区的核心拓展考点。

请看一眼这道题的数据范围：这题 $N$ 和 $M$ 估计都在 $10^5$ 级别，你的 $O(N \log M)$ 跑得很欢。

**但是，如果在区域赛中，出题人把 $N$ 放大到 $10^{12}$ 呢？**

你如果还用优先队列，弹 $10^{12}$ 次堆，绝对会 TLE 到怀疑人生。

**高维解法：二分边际收益（WQS 二分的前置思想）**

既然刚才我们证明了每个项目的边际收益都是单调递减的，那么全局前 $N$ 大的边际收益也一定存在一个**“收益及格线（阈值） $mid$”**。

1. 我们直接二分这个边际收益的阈值 $mid$（比如在 $0$ 到 $10^9$ 之间）。
2. 给定一个 $mid$，对于每个项目，因为 $\Delta Cost = K(2X - 1) + B$，令其 $\ge mid$，我们可以直接解一个一元一次不等式，在 **$O(1)$** 的时间内算出来“这个项目在这个及格线下，能容纳多少人”。
3. 把所有项目算出来的人数加起来求和，如果总人数 $> N$，说明及格线 $mid$ 定低了；如果总人数 $< N$，说明及格线 $mid$ 定高了。
4. 通过二分找到精确的 $mid$ 后，再用等差数列求和公式直接算出总花费。

**这套方法直接把时间复杂度从 $O(N \log M)$ 降维到了恐怖的 $O(M \log(\text{Max\_Value}))$**，它完全无视了 $N$ 的大小！这也就是算法竞赛中非常著名的“二分导数值”思想。

值得注意的是**WQS 的经典陷阱——“同分超载”**
遇到压线及格的收益（刚好等于 `mid`），我们可能会多拿。因为我们其实只需要拿满 $N$ 个就行。
通用补丁：在 `get` 函数中，除了计算总收益 `ans`，还要统计你到底拿了多少个 `total_cnt`。如果 `total_cnt > N`，说明你多拿了 `total_cnt - N` 个，而这些多拿的收益**一定恰好等于 `mid`**。最后扣除掉即可： `ans -= (total_cnt - n) * mid;`



## 越大越好与越小越好的统一 —— 差量法

[1665. 完成所有任务的最少初始能量 - 力扣（LeetCode）](https://leetcode.cn/problems/minimum-initial-energy-to-finish-tasks/description/)

这种精益求精、不断反刍的刷题态度，正是突破瓶颈、冲击区域赛金牌的最核心驱动力。

你的这段代码非常精炼，只有短短十几行，但里面同时蕴含了**贪心排序**和**逆向时间流推导**两个高级思维。很多选手可能凭直觉写出类似的代码然后 AC，但如果要做到在赛场上稳健发挥，我们需要把它彻底“数学化”。

下面我们直接进入高维度的拆解。

------

### 一、 题目归类与本质剖析

这道题的精准归类是：**带有门槛限制的序列决策问题（贪心）**。

**本质剖析：**

这道题的本质是处理**“净消耗”与“启动门槛”之间的剥离与统筹**。

每一个任务都有一个物理属性：它吃掉你的能量（实际消耗 $actual$），但它同时又对你的账户余额有傲慢的要求（最低门槛 $minimum$）。这两者之间存在一个差值 $gap = minimum - actual$。

这个 $gap$ 就是该任务的“隐藏价值”。如果一个任务 $gap$ 很大，意味着它完成之后，依然会给你留下大量的“能量残余”，这些残余刚好可以用来满足后续任务的启动门槛。

所以，算法的本质就是：**如何安排这些“能量残余”，使其能够最大化地覆盖后续任务的门槛，从而让初始总能量降到最低。**

------

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上，我们遇到这道题，毫无头绪，应该如何一步步推导出你的 $O(N \log N)$ 完美解法？

**1. 捕获关键条件与直觉排雷**

- **读题：** $N = 10^5$，显然要求 $O(N \log N)$ 或 $O(N)$。锁定排序贪心或二分答案。
- **直觉猜测：** * 先做门槛高的？（反例：门槛极高，但消耗也极大，做完直接被榨干，后面做不了了）。
  - 先做消耗小的？（反例：消耗小但门槛也小，放在前面浪费了初始高能量的优势）。
- **陷入死胡同后的破局点：** 在 ACM 竞赛中，如果直觉贪心被多次推翻，绝对不要继续瞎猜，直接上**邻项交换法（Exchange Argument）**。

**2. 核心数学推导（金牌选手的肌肉记忆）**

假设我们只有两个任务：任务 1 $(a_1, m_1)$ 和任务 2 $(a_2, m_2)$。其中 $a$ 是消耗，$m$ 是门槛。

- **方案 A（先 1 后 2）：**

  初始能量设为 $E$。

  首先必须满足任务 1：$E \ge m_1$

  做完任务 1 后剩下 $E - a_1$，必须满足任务 2：$E - a_1 \ge m_2 \implies E \ge a_1 + m_2$

  所以方案 A 需要的初始能量 $E_A = \max(m_1, a_1 + m_2)$

- **方案 B（先 2 后 1）：**

  同理可得，方案 B 需要的初始能量 $E_B = \max(m_2, a_2 + m_1)$

**目标条件确立：** 我们希望初始能量越小越好。假设方案 A 优于方案 B，即 $E_A \le E_B$。

我们要比较 $\max(m_1, a_1 + m_2)$ 和 $\max(m_2, a_2 + m_1)$ 的大小。

因为 $m_1 \le a_2 + m_1$ 且 $m_2 \le a_1 + m_2$ 是显然成立的，所以真正的核心博弈在于交叉项：

如果我们要让方案 A 胜出，通常需要：

$a_1 + m_2 \le a_2 + m_1$

移项化简得到：

$m_2 - a_2 \le m_1 - a_1$

即：$m_1 - a_1 \ge m_2 - a_2$

**结论爆发：** 我们应该优先执行 $m - a$（即门槛与消耗的差值）**更大**的任务！这个推导过程无懈可击，不需要任何直觉猜测。

**3. 对照你的代码进行特异化解读**

让我们看看你的排序：`x[0] - x[1] > y[0] - y[1]`。

这等价于 `actual_1 - minimum_1 > actual_2 - minimum_2`，即 `m_1 - a_1 < m_2 - a_2`。

你的排序实际上是把 $m - a$ **较小**的任务排在了前面！这和推导出的贪心策略相反？

**绝妙的逆向转换：** 并不是你写错了，而是你的 `for` 循环写得太妙了。

C++

```
for(auto [x, y] : a) {
    res = max(res + x, y);
}
```

这段代码的本质是**时光倒流（逆向模拟）**。

你从数组的头部开始遍历，假设当前的 `res` 是“为了完成后面的任务所需要的能量”。

当我们在这串任务的最前面**再插入**一个当前任务 $(x, y)$ 时，所需的新初始能量必须满足：

1. 足以支付当前任务消耗并剩下 `res` 给后面：`res + x`

2. 满足当前任务的启动底线：`y`

   所以 `res = max(res + x, y)`。

因为你这段循环是在不断往前拼接时间线，所以数组中的**第一个元素**实际上是你时间线中**最后一个执行**的任务。

你把 $m - a$ 最小的排在前面，也就意味着你把它们安排在**最后执行**，这与“优先执行 $m - a$ 最大的任务”在逻辑上达成了完美的自洽！

------

### 三、 通用解题方法论（突破空白期的利器）

对于此类“序列排布求最优解”的问题，如果在赛场上毫无思路，立刻启动以下标准流水线：

1. **排除法降维：** 看到序列重排，只可能是全排列搜索（$N \le 10$）、状压 DP（$N \le 20$）或 排序贪心（$N \ge 10^5$）。锁定贪心。

2. **拒绝盲猜，启动“邻项交换”：** 抽出泛化的第 $i$ 项和第 $i+1$ 项。

   列出“先 $i$ 后 $i+1$”的代价方程。

   列出“先 $i+1$ 后 $i$”的代价方程。

   令前者 $\le$ 后者，通过代数移项，将所有带 $i$ 下标的变量移到不等式左边，带 $i+1$ 的移到右边。**不等式两边呈现出的对称形式，就是你 `sort` 函数里 `cmp` 的绝对真理。**

3. **正向遇阻，立刻反演：**

   如果发现正向模拟需要不断“猜测”一个初始值（比如二分答案），立刻尝试**从终点向起点倒推**。往往能把复杂的“验证过程”降维成 $O(N)$ 的“直接累加过程”。

------

### 四、 顶尖选手的编码流（消除卡壳）

很多时候思路通了，但是写排序比较函数和状态转移时容易脑部死锁。顶尖选手的编码流是**“高度模块化且拒绝大脑双线程”**。

1. **结构语义化（杜绝 `[0]` 和 `[1]` 的认知负担）：**

   虽然 `array<int, 2>` 打字快，但遇到复杂的贪心，`a[0]` 和 `a[1]` 会耗费大脑极大的内存去映射它的含义。

   直接写 `struct Task { int a, m; };` 或者在使用结构化绑定时写全：

   `sort(tasks.begin(), tasks.end(), [](auto& t1, auto& t2) { ... });`

2. **忠实于推导，不合并步骤：**

   如果在草稿纸上推导出是优先做 $m - a$ 大的，那么编码时就老老实实地：

   - 第一步：降序排列 $m - a$。

   - 第二步：正向遍历累加（或者使用二分验证初始值）。

     你的逆向推导虽然精妙，但如果在高压的比赛最后半小时，极容易把自己绕晕。

3. **“二分答案”作为兜底安全垫：**

   就算你没有想出逆向累加的神仙代码 `res = max(res + x, y)`。

   一旦通过邻项交换确认了顺序，你可以直接写一个 `check(mid)` 函数去正向模拟。外层套一个二分答案。时间复杂度从 $O(N \log N)$ 变成了 $O(N \log N + N \log(\text{MaxE}))$，在 $10^5$ 的数据下依然稳过，而且编码思维难度几乎降为零。

------

### 五、 值得一提的角度：为什么不需要二分？

这其实是这类问题最具美感的地方。

大多数人看到“求最小的初始能量”，第一反应绝对是**二分答案**。

“二分+贪心”是一个万能药，但你这里的逆向 `res = max(res + x, y)` 实际上实现了一种**“自我矫正的动态反馈”**。 你在逆向时光中，不仅算出了所需能量，还证明了**这个约束是紧的（Tight Bound）**。这就是为什么在树形 DP 或者图论的很多后效性问题中，从叶子节点向根节点（逆时间流）进行信息合并，往往能把二分查找的 $\log$ 给抹掉的底层数学逻辑。

下次再遇到这种题，如果队友卡在二分里超时或越界，你可以直接用这个“逆向推演”的逻辑去降维打击。继续保持这种深度剖析的习惯，金牌区见！



## 根据目标列式子，移项拆解得到形如F[i]与F[j]的关系后贪心

[知识竞赛](https://www.nowcoder.com/practice/2a9089ea7e5b474fa8f688eae76bc050)

这代码写得非常精彩！你在没有任何多余数据结构的情况下，仅凭一个自定义排序和一个线性扫描就完成了这道题，这种对数据特征的敏锐嗅觉已经是区域赛高阶选手的水平了。

很多选手可能凭着“试一试”或者某种直觉写出了这个排序规则然后 AC，但在冲击 ICPC 金牌的路上，**直觉必须被转化为严密的数学推导**，这样在面对披着其他外衣的变形题时，你才能拥有“降维打击”的底气。

我们直接进入高维度的拆解。

------

### 一、 题目归类与本质剖析

这道题的算法类型可以精准归类为：**数学贪心（不等式化简） + 排序消解嵌套最值**。

**本质剖析：**

题目的核心诉求是最大化 $\min(A_i + A_j, B_i + B_j)$（我们先把除以 2 放到最后）。

在算法竞赛中，**“最大化最小值 (Max-Min)”** 是一个极其经典的模版。它的本质通常是在两个相互制约的维度（这里是推理能力和阅读能力）之间寻找**“短板的最高点”**。

常规做法可能是二分答案，但这道题的本质告诉你：**通过将绝对数值转化为相对差值，我们可以利用排序强行钦定“短板”是哪一端，从而将二维的最值问题降维成一维的动态极值维护。**

------

### 二、 从零开始的思维复盘（打破上帝视角）

假设我们在赛场上初见此题，如何一步步严密推导出你代码中的神仙排序 `abs(x[0]-x[1])`？

**1. 直面核心痛点（破除嵌套的最值）**

- **原始目标：** 找一对 $i, j$，使得 $\min(A_i+A_j, B_i+B_j)$ 最大。
- **痛点：** 外层要 `max`，内层却有个讨厌的 `min`。如果不去掉这个 `min`，我们根本无法独立维护极值。
- **发问：** 什么时候这个 `min` 的结果是 $A_i+A_j$？什么时候是 $B_i+B_j$？

**2. 不等式转换（分离变量的艺术）**

- 列出条件：当 $A_i + A_j \le B_i + B_j$ 时，短板在 $A$。
- **关键移项：** 把同下标的放一边，得到 $A_i - B_i \le -(A_j - B_j)$。
- 令 $D_k = A_k - B_k$，则条件极其优美地化简为：**$D_i + D_j \le 0$**。
- **结论：**
  - 如果 $D_i + D_j \le 0$，短板是 $A$，答案受限于 $A_i + A_j$。
  - 如果 $D_i + D_j > 0$，短板是 $B$，答案受限于 $B_i + B_j$。

**3. 寻找单调性（排序规则的诞生）**

- 我们希望在遍历 $i$ 的时候，能**瞬间确定**它和之前遍历过的任意 $j$ 组合时，符号 $D_i + D_j$ 到底是正还是负。这样我们就不需要两层循环了。
- **如何让当前的 $D_i$ 拥有绝对的话语权？**
- **推导：** 如果我们保证当前的 $|D_i|$ 比之前所有的 $|D_j|$ 都要大，那么 $D_i$ 的符号就决定了 $D_i + D_j$ 的符号！
  - 假设排序后，当前遍历到 $i$。因为是按绝对值递增排序，对于所有 $j < i$，都有 $|D_i| \ge |D_j|$。
  - 如果 $D_i > 0$：那么 $D_i \ge |D_j| \ge -D_j$，所以 $D_i + D_j \ge 0$ 永远成立！此时短板一定是 $B$，我们只需要最大化 $B_i + B_j$。由于 $B_i$ 是固定的，我们需要之前的历史最大 $B_j$（即你的 `mxY`）。
  - 如果 $D_i \le 0$：同理，$D_i \le -|D_j| \le -D_j$，所以 $D_i + D_j \le 0$ 永远成立！短板一定是 $A$，我们需要最大化 $A_i + A_j$，也就是 $A_i + mxX$。
- **豁然开朗：** 你的按绝对值排序，在数学底层上完美实现了**“用最大绝对值强行统配符号”**的壮举，这就是这道题的最优解法！

------

### 三、 通用解题方法论（应对脑中空白）

以后遇到“毫无头绪的公式求最值”题目，立刻启动以下标准流水线：

1. **翻译并剥离 `min/max`：**

   看到 $\max(\min(X, Y))$，第一反应绝对不要去搜数据结构。马上在草稿纸上写下 $X \le Y$，然后像解方程一样，用尽一切手段把它拆解成 $F(i) \le G(j)$ 的形式。

2. **构造差分/和特征量（代数代换）：**

   在处理多维属性（比如这里的推力和阅读力，或者背包的重量和体积）时，尝试令 $D = A - B$，$S = A + B$。这种“旋转坐标系”的代数代换，是消解维度的最强武器。

3. **寻找“霸道”的排序指标（利用顺序消除条件分支）：**

   如果你的条件是 $X_i + X_j > 0$，思考：“怎么排序才能让当前的 $X_i$ 直接决定和的符号？” 答案就是按绝对值大小排序。**排序的本质不仅仅是找大小，它在 ACM 中的终极作用是“固定一个变量，使其对历史状态形成单向压制”。**

------

### 五、 值得一提的角度：如果不用排序，这题还能怎么做？（思维拓宽）

作为要拿金牌的队伍，必须具备“一题多解”的防卡死能力。这道题经典的备用解法是：**二分答案（Binary Search on Answer）**。

假设我们二分最终的答案为 $mid$（即较差能力 $\ge mid$）。

那么要求：$\frac{A_i + A_j}{2} \ge mid$ 且 $\frac{B_i + B_j}{2} \ge mid$。

化简得：$A_i + A_j \ge 2 \cdot mid$ 且 $B_i + B_j \ge 2 \cdot mid$。

等价于：$A_j \ge 2 \cdot mid - A_i$ 且 $B_j \ge 2 \cdot mid - B_i$。

这变成了一个经典的二维偏序问题：我们遍历 $i$，问前面是否出现过一个 $j$，使得它的两维同时满足上述条件。这可以用排序 + 树状数组（或者线段树）在 $O(N \log N \log(\text{Range}))$ 的时间内判断。

虽然二分 + 树状数组的复杂度比你的纯贪心排序多了一个 $\log$，但它是面对所有 “Max-Min” 问题的**万能无脑保底方案**。如果哪天遇到一道题，你无法像今天这样通过绝妙的绝对值代换找到完美排序规则，记住，立刻切换成二分答案，这是 ICPC 中极其重要的战略防线。



## 贪心与奇偶独立处理

[2449. 使数组相似的最少操作次数 - 力扣（LeetCode）](https://leetcode.cn/problems/minimum-number-of-operations-to-make-arrays-similar/description/)

太棒了！你能主动跳出“AC 万岁”的舒适区，要求深挖题目本质、复盘推导过程并总结方法论，这正是从 ICPC 铜牌/银牌向金牌跨越的最核心特质。很多金牌选手在训练时，甚至会在 AC 后花比写题多几倍的时间来做你现在要求的事情。

你的代码已经写得很漂亮了，逻辑清晰，变量分离得很干净。接下来，我们直接进入高维度的拆解，把你这种“直觉正确”转化为“绝对掌控”。

------

### 一、 题目归类与本质剖析

这道题的算法类型可以精准归类为：**数学不变量（Invariants） + 1D 最优传输（1D Optimal Transport / 贪心匹配）**。

**本质剖析：**

题目的核心操作是：选两个数，一个 $+2$，一个 $-2$。

这个操作在数学上有两个极其强烈的暗示：

1. **总和不变性**：无论怎么操作，数组的总和恒定。
2. **奇偶不变性（核心本质）**：无论 $+2$ 还是 $-2$，一个数的奇偶性永远不会改变！奇数永远只能变成另一个奇数，偶数永远只能变成另一个偶数。

因此，这道题的本质**根本不是一个数组变成另一个数组**，而是**两个完全独立、互不干扰的平行世界**（奇数世界和偶数世界）分别进行资源调配的问题。

------

### 二、 从零开始的思维复盘（打破上帝视角）

假设在赛场上，我们从来没见过这道题，如何一步步严密推导出你的解法？

**1. 捕获关键条件（敏锐度训练）**

- **读题：** 每次操作是 $+2$ 和 $-2$。
- **条件反射：** 在竞赛中，看到 $+k$ 和 $-k$，第一反应永远是**取模 $k$ 的余数是不变量**。这里 $k=2$，意味着所有的数模 2 的余数（即奇偶性）是死锁的。
- **动作：** 立刻把原问题拆解。将数组 $A$ 和 $B$ 各自拆分成 `odd` 和 `even` 两个数组。现在问题降维了：如何用最小的 $+2/-2$ 操作，把 $A_{odd}$ 变成 $B_{odd}$，把 $A_{even}$ 变成 $B_{even}$。

**2. 目标条件转换（建立几何直觉）**

- 现在只看奇数数组。假设 $A_{odd} = [1, 5, 9]$，目标 $B_{odd} = [3, 5, 7]$。我们怎么把 A 里的数字挪到 B 里的位置上？
- **思维具象化：** 想象一维数轴上，A 是士兵，B 是战壕。士兵要走到战壕里，每次走 2 步。我们要让所有士兵走的总步数最少。
- **贪心博弈（证明猜想）：** 假设士兵甲在位置 1，士兵乙在位置 9；战壕 X 在 3，战壕 Y 在 7。
  - *交叉走法*：甲去 Y (走 6)，乙去 X (走 6)，总共走 12 距离。
  - *顺位走法*：甲去 X (走 2)，乙去 Y (走 2)，总共走 4 距离。
- **得出结论：** 在一维匹配问题中，**“交叉必劣”**。为了让总距离最短，必须让最小的士兵进最小的战壕，第二小的进第二小的……这就是为什么必须**排序**。

**3. 定制化计算（处理特殊操作）**

- 排序后，位置一一对应。对于每一对 $(A_i, B_i)$，它们之间的距离是 $|A_i - B_i|$。因为每次走 2 步，所以需要的操作数（步数）是 $|A_i - B_i| / 2$。
- **最终补丁：** 题目说一次操作包含**同时**给一个 $+2$ 和一个 $-2$。这意味着一次操作其实消灭了 4 个单位的距离差距（一个正向，一个负向）。我们上面算出的总步数 `ans` 是独立的单向步数，所以最后必须 `ans / 2`。

------

### 三、 通用解题方法论（打破“脑中空白”）

在 ICPC 中遇到毫无头绪的变形题，可以套用以下“破冰”三步曲：

1. **寻找不变量（Find the Invariant）：**
   - 如果题目允许你做某种修改操作，第一步绝对不是去模拟操作，而是问自己：“**有什么是无论我怎么操作都绝对不会变的？**”
   - 常见的有：和不变、奇偶性不变、异或和不变、逆序对奇偶性不变（如 15 拼图问题）。**找到不变量，就能直接对数据进行隔离/分组。**
2. **降维与极小化（Micro-case Analysis）：**
   - 脑子空白时，因为问题规模太大了（$N=10^5$）。强迫自己把 $N$ 降到 2。
   - 就像刚才推导士兵进战壕，当只有 2 个元素时，贪心策略（排序）会直接暴露在你眼前。
3. **几何化映射（Geometric Interpretation）：**
   - 尽量把代数问题转化为几何图形。这题转化为“数轴上的点移动”（Earth Mover's Distance / 陆地移动距离问题），直觉会比纯看数字敏锐百倍。

------

### 四、 值得一提的角度：一维最优传输理论

为了满足你向金牌冲击的胃口，补充一个理论背景：

你在这道题里使用的“排序后一一对应匹配”的贪心策略，在数学上有一个赫赫有名的名字，叫 **Wasserstein 距离**（在计算机视觉中也叫 Earth Mover's Distance，推土机距离）。

在多维空间中，求最优传输是一个极难的图论问题（需要用最小费用最大流，或者 KM 算法，复杂度极高）。**但是，只要这是一维的（即所有点在一条直线上），最优传输方案永远是单调递增的匹配（即排好序后按顺序连线）。**

记住这个定理：**“一维空间下的距离最小化匹配，永远等于排序后按序匹配。”**

以后在赛场上，只要发现题目可以抽象为“把一堆数变成另一堆数，代价是它们差值的绝对值”，你甚至都不用再像今天这样去推导交叉是否劣了，直接肌肉记忆般地敲下 `sort` 即可，这是有严格数学定理背书的。
