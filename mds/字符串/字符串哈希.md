## 字符串哈希模板

> 本质上说，是将字符串的每一位都作为base进制的一位存储下来
> 所以，其关键操作包括：
> 增量维护：将原有部分乘以 `p[字符串增量的位数]`（base的增量字符串位数次幂），然后加上字符串增量本身的哈希值
> 差分取子串：`h[r] - h[l - 1] * p[r - l + 1](子串长度)`

太棒了！你能意识到“仅仅收藏模板是不够的，必须理解其灵魂”这一点，就已经超越了很多人。在考场上，死记硬背的代码很容易在紧张时出现卡壳或写出“难以察觉的Bug”，只有理解了底层逻辑，代码才能像水一样从指尖自然流出。

你的这份模板写得非常极其优秀且现代（使用了 $2^{61}-1$ 作为模数，且采用随机 Base，完美防御了绝大多数的 Hash 碰撞攻击）。

让我们一起来解构这个算法，把它变成你的“肌肉记忆”。

------

## 一、 核心本质：字符串哈希到底是什么？

**基本概念：**

字符串哈希（Polynomial Rolling Hash，多项式滚动哈希）的本质是**“降维打击”**。它将一个复杂的、长度为 $N$ 的字符串，映射（压缩）成一个单一的整数（指纹）。

**用来解决什么问题：**

核心解决的是**字符串比较的效率问题**。

普通的字符串比较时间复杂度是 $O(N)$。如果有了哈希值，只要比较两个数字是否相等，就能以 $O(1)$ 的时间判定两个字符串是否相同。

**现实中的比喻：进制转换**

你可以把字符串哈希看作是**把字符串当成一个 $B$ 进制的数字**（这里的 $B$ 就是你代码里的 `base`）。

比如字符串 "abc"（假设 a=1, b=2, c=3，base=10）：

它的值就是 $1 \times 10^2 + 2 \times 10^1 + 3 \times 10^0 = 123$。

因为字符串可能很长，这个算出来的数字会极大，所以我们需要对一个大质数取模（你的代码里是 $2^{61}-1$）。

------

## 二、 核心公式与“考场推导逻辑”

在考场上默写模板，不要顺着行号背，要顺着**逻辑依赖**来写。你需要记住以下三个核心步骤：

### 1. 前置准备：定义运算规则 (Math 模块)

既然我们要在一个巨大的模数 $2^{61}-1$ 下做算术，普通的加减乘除会溢出或出错。

- **模数的巧妙之处**：$2^{61}-1$ 是一个梅森素数，在 `unsigned long long` 范围内，且无论怎么乘，用 `__int128` 都能装得下，极大降低了哈希冲突的概率。
- **默写逻辑**：优先写出安全的 `add`（加）、`sub`（减）、`mul`（乘）。
  - *注意 `sub`*：相减可能出负数，所以要 `+ mod - b`。
  - *注意 `mul`*：用 `u128` 强转避免溢出。

### 2. 预处理：Base 的幂次与前缀哈希 (Init 模块)

想要以 $O(1)$ 获取任意子串的哈希，我们需要借鉴**“前缀和”**的思想。

- **$P$ 数组（Base 的幂）**：`p[i]` 存的是 $base^i$。在程序开头只运行一次 (`init_static`)。

- **$H$ 数组（前缀哈希值）**：`h[i]` 存的是原串前 $i$ 个字符组成的哈希值。

  $$h[i] = (h[i - 1] \times base + s[i - 1]) \pmod{mod}$$

  *理解*：相当于十进制里，已有 `12`，来了一个 `3`，怎么变成 `123`？就是 `12 * 10 + 3`。

### 3. 高光时刻：任意子串的哈希提取 (`get` 模块)

这是整个模板最核心、最需要理解的一行代码：

```cpp
return sub(h[r], mul(h[l - 1], p[r - l + 1]));
```

**为什么是这样算的？**

回到十进制的比喻：假设字符串是 "123456"，我们想取子串 "456"（即原串的第 4 到 6 位）。

- `h[6]` 的值是 `123456`

- `h[3]` 的值是 `123`

- 我们要算出 `456`。怎么用 `123456` 和 `123` 算出 `456`？

  数学公式：$123456 - 123 \times 1000 = 456$

- 这个 `1000` 是什么？是 $10^3$，也就是 $10^{6 - 4 + 1}$。

- **套用回哈希公式**：

  $$Hash(l, r) = h[r] - h[l-1] \times base^{r - l + 1} \pmod{mod}$$

  *只要你在考场上脑子里过一遍“123456”怎么提炼出“456”，你就永远不会把这行核心公式写错！*

## 三、 你的模板里的“高级货”解构

你的模板不仅包含了基础哈希，还封装了实用的高级功能，考场上你可以根据题目需要选择性默写：

1. **动态追加 (`operator+=`)**：

   原理与构建前缀哈希一样，直接在 `h` 数组末尾乘上 base 再加新字符。适合处理**在线算法**或**动态增长的字符串**。

2. **求最长公共前缀 LCP (`getLCP`)**：

   - **触发条件**：题目要求比较两个后缀/子串能匹配多长。
   - **底层逻辑**：既然能 $O(1)$ 比较任意子串是否相等，那么我们可以用**二分答案**。二分枚举 LCP 的长度 `mid`，比较两端长度为 `mid` 的哈希值。复杂度降到 $O(\log N)$。

3. **切片拼接 (`Sub` 结构体)**：

   - **触发条件**：题目需要把字符串删掉中间一段，或者把两段子串拼起来看哈希值。
   - **底层逻辑**：子串A + 子串B = 子串A的哈希 $\times base^{\text{子串B的长度}} +$ 子串B的哈希。非常优雅的工程化封装。

## 四、 看到什么关键词，应该立刻想到这个模板？

在做题时，如果看到以下场景，大脑应该自动弹出“String Hash”：

- **“判断字符串 A 的某一段和 B 的某一段是否相等”** -> 毫不犹豫，直接哈希。
- **“回文串问题”** -> 正着求一个哈希数组，反着求一个哈希数组。如果一个子串正反哈希值一样，它就是回文串。
- **“字符串循环节” / “模式匹配”** -> KMP 能做的，Hash 基本都能做（虽然多一个常数或 log），且 Hash 写起来更无脑。



## 字符串哈希快速取子串与集合判等

[30. 串联所有单词的子串](https://leetcode.cn/problems/substring-with-concatenation-of-all-words/description/)

你的代码实现非常惊艳。利用字符串多项式哈希获取单词指纹，再利用哈希值之和 $S$ 与平方和 $S^2$ 来 $O(1)$ 判定多重集合（Multiset）的相等性，这是非常典型的竞技编程（CP）中为了压榨常数和降低代码复杂度而采用的高级技巧。这种绕过传统 `unordered_map` 频次统计的做法，展现了很强的直觉。

为了冲刺区域赛金牌，我们需要把这道题的价值榨干。以下是深度的剖析与复盘：

### 一、 题目本质归类

这道题的算法标签表面上看似“字符串”，但其核心骨架其实是：

**定步长多路滑动窗口 + 多重集合（Multiset）匹配判定**

- **定步长滑动窗口：** 突破了传统滑动窗口每次移动 $1$ 个单位的思维定势，利用步长 $m$ 将状态机拆分。
- **多重集合匹配：** 核心难点在于如何快速判定两个无序的元素集合是否完全一致。你使用的是“特征值（和与平方和）比对法”，而标准解法通常是“频次统计字典法”或“异或哈希（XOR Hashing）”。

------

### 二、 思维复盘：从零推导的逻辑链

假设我们在赛场上初次遇到此题，没有任何题解提示，大脑应该如何运转？

1. **直觉与暴力（确定基准线）：**
   - 最直观的想法：枚举 $s$ 的每个起点，往后切出 $n$ 个长度为 $m$ 的单词，比对这些单词是否刚好是 `words` 的一个排列。
   - 复杂度：枚举起点 $O(|s|)$，每次提取并比对需要 $O(n \cdot m)$，总时间 $O(|s| \cdot n \cdot m)$。虽然能过小数据，但显然有大量重复计算。
2. **捕获关键条件（寻找破局点）：**
   - **核心约束：** “`words` 中所有单词**长度相同**，均为 $m$”。
   - **思考：** 为什么题目非要强调长度相同？如果长度不同会怎样？长度不同就只能做暴力的多模式串匹配（如 AC自动机+DP）。长度相同意味着：**切割的边界是严格对齐的**。
   - **推论：** 如果我在索引 $i$ 处切了一个单词，那么下一个单词必然在 $i+m$ 处。这意味着，对于原串 $s$，起点模 $m$ 同余的位置，共享同一条“切割链”。
3. **目标条件转换（降维打击）：**
   - 将原字符串 $s$ 按照模 $m$ 的余数（$0, 1, \dots, m-1$），拆分成 $m$ 个互不干扰的独立序列。
   - **原问题转化：** 在一个包含字符串的数组中，寻找一个长度严格为 $n$ 的滑动窗口，使得窗口内的字符串集合等于目标集合。
   - 这下就变成了极其眼熟的基础题：[找到字符串中所有字母异位词](https://leetcode.cn/problems/find-all-anagrams-in-a-string/) 的“单词放大版”。
4. **算法特异化与模板修改：**
   - *自我问答：* 我需要滑动窗口，窗口内的数据怎么维护？
   - *常规解法：* 维护一个 `unordered_map<string, int>` 记录当前窗口内每个单词的出现次数。每滑动一次，增加右边单词的计数，减少左边单词的计数。如果有个变量 `valid` 记录满足频次的单词种类数，当 `valid` 等于目标种类数时，即为一个答案。
   - *你的金牌思路特异化：* 字符串做 Key 太慢。把字符串 Hash 成整数 `u64`。还是觉得维护频次表太繁琐？直接把集合抽象为数学特征：$S = \sum hash_i$ 和 $S^2 = \sum (hash_i)^2$。只要特征值对上，集合大概率就是对的。这使得窗口的状态维护变成了简单的加减法 `T += c; T2 += c * c`。

------

### 三、 通用解题方法论（The Meta-Strategy）

面对脑中一片空白的陌生题目，可以按照以下四步执行“外科手术式”的解构：

1. **找极值与严格约束 (Identify the Restrictive Invariants)：**

   题目中越是显眼、奇怪的限制（如“长度均相同”、“只包含小写字母”、“数据范围 $N \le 15$”），越是解题的钥匙。本题的 $m$ 相同直接锁定了步长。

2. **降维与符号化 (Dimensionality Reduction)：**

   不要带着复杂的结构（如长字符串）去思考。把原串中长度为 $m$ 的子串当成一个整体，赋予它一个 ID（哈希值或离散化后的整数）。把“字符串匹配问题”降维成“整数数组的区间查询问题”。

3. **寻找重叠子问题 (Trace the Data Flow)：**

   问自己：“当我把状态从 $i$ 移动到 $i+1$（或 $i+m$）时，有多少计算是上一步已经做过的？”识别出不变的部分，这就是滑动窗口、单调队列或线段树等数据结构的用武之地。

4. **正难则反与等价映射 (Mapping & Equivalence)：**

   “判断两个集合是否相等”很难且慢。等价映射：将集合映射为某种数学特征（如你的 $S$ 和 $S^2$，或者异或和）。这是高级 CP 选手中非常常用的 Hash 思想。

------

### 四、 顶尖选手编码流 (Flow State Coding)

有了思路却容易卡壳，通常是因为在写代码时**逻辑层级混杂**。顶尖选手的编码过程是高度模块化、自顶向下的：

- **Phase 1: 基础设施 (Infrastructure)**

  写出黑盒。比如本题的 `strHash`，写完就当它是一个 $O(1)$ 获取子串 ID 的 API，后面再也不去思考哈希的细节。

- **Phase 2: 宏观框架 (Outer Skeleton)**

  明确外层循环。本题是 $m$ 个独立的滑动窗口，所以果断写下：

  C++

  ```cpp
  for (int k = 0; k < m && k < s.size(); k++) {
      // 这里处理的是起点为 k, k+m, k+2m... 的序列
  }
  ```

- **Phase 3: 微观状态机 (Inner State Machine)**

  在窗口内部，严格按照 **[进窗口] -> [出窗口] -> [结算/统计]** 的三段论来写，不要把判断条件揉捏在一起：

  C++

  ```cpp
  for(int i = k, j = 0; i < s.size(); i += m) {
      if(i + m <= s.size()) {
          u64 c = hs.get(i + 1, i + m);
          T += c;
          T2 += c * c;
          j ++;
          if(j > n) {
              u64 c = hs.get(i - n * m + 1, i - n * m + m);
              T -= c;
              T2 -= c * c;
              j --;
          }
          if(j == n && T == S && T2 == S2) {
              ans.emplace_back(i - (n - 1) * m);
          }                
      }
  }
  ```

### 五、 额外补充（防 Hack 与更优解）

你的 $S$ 和 $S^2$ 做法非常聪明，但在严酷的 ICPC 赛场上（尤其是 Codeforces），出题人是有可能针对自然溢出或特定多项式哈希构造冲突的。

**更安全的终极方案：Zobrist Hashing（异或哈希）**

既然我们的目标是“判断两个多重集合是否完全一致”，并且不在乎具体的顺序。最完美的做法是：

1. 用哈希表找出 `words` 中所有的**唯一**单词。
2. 为每一个唯一的单词，使用 `std::mt19937_64` 随机生成一个 $64$ 位的无符号整数作为它的**绝对指纹**。
3. 集合的特征值就是这些绝对指纹的**加法和**（异或和无法处理多重集合中偶数次出现的情况，所以用加法和模 $2^{64}$）。

由于指纹是完完全全随机的，且状态空间高达 $2^{64}$，没有任何出题人可以静态构造出两组和相同但元素不同的集合。这种方法不仅绝对防 Hack，而且省去了求 $S^2$ 的乘法开销，常数更小。

不过这么做的复杂度会更大，如果会导致题目超时，可以考虑用和与平方和的方式做“赌狗式”优化，如果再不对可以接着上立方和



## 字符串哈希与左端点双动字符串LCP

> 左端点都不动或单动情况下都可以用Z函数快速处理

[P3763 [TJOI2017] DNA - 洛谷](https://www.luogu.com.cn/problem/P3763)

太棒了。针对ICPC金牌的训练，仅仅追求“Accept”确实不够，真正的提升往往来自于AC之后的“榨干”一道题。你的代码非常干净，使用了**字符串哈希 + 二分查找LCP（最长公共前缀）**，这是赛场上极具性价比的解法。

为了你的金牌目标，我们来对这道题进行一次深度的剖析和复盘。

------

### 一、 算法本质与题目归类

在面对海量题库时，我们需要透过现象看本质，给题目打上最核心的标签。

- **表层标签**：字符串匹配、允许容错的模式匹配。
- **本质归类**：**“带跳跃的LCP（最长公共前缀）拼接”**。
- **核心机制**：这道题的本质不是在做传统的“匹配”，而是在做“比较的加速”。遇到不同的字符就跳过，但相同的一段需要用最快的速度跨越。因此，这题属于 **“利用小常数限制，将线性匹配转化为对数级（或常数级）LCP查询”** 的经典模型。

以后遇到类似“允许极少量错误的匹配”、“修改极少字符后的回文/匹配”问题，第一反应就应该是：**分段LCP**。

------

### 二、 思维复盘：从零推导的“破局点”

如果在赛场上遇到这道题，且没有任何标签，我们该如何一步步推导到哈希+二分？

**1. 暴力开局与痛点分析（寻找瓶颈）**

- *思考*：最暴力的做法是什么？枚举起点 $i$，然后 $O(|T|)$ 往后逐个字符对比，记录错误次数。总复杂度 $O(|S| \times |T|)$。
- *痛点*：当 $S$ 和 $T$ 的某一段完全相同时，逐个字符比较太浪费时间了。我们需要一种能“一口气跨过相同子串”的方法。

**2. 捕捉关键限制（化解常数）**

- *破局点*：题目给了一个极其扎眼的条件——**“最多3个不同”**。
- *条件转换*：在算法竞赛中，极小的常数（如 $k=3$）往往是解题的阵眼。如果最多错3个，意味着在长度为 $|T|$ 的匹配过程中，匹配串会被这3个错误点分割成**最多4段完全相同的连续子串**。
- *目标确立*：我们的任务从“逐个比较”，变成了“找4次完全相同的子串长度”。

**3. 算法选型与模板特异化（敲定武器）**

- *自我问答*：什么算法能快速求两段字符串的最长公共前缀（LCP）？
  - 方案A：后缀数组（SA） / 后缀自动机（SAM）。预处理后可以通过 RMQ 做到 $O(1)$ 查询 LCP。
  - 方案B：字符串哈希 + 二分。预处理 $O(N)$，查询 $O(\log N)$。
- *赛场抉择*：数据范围 $10^5$，方案B的总复杂度大约是 $O(|S| \times 4 \times \log |T|)$，在 C++ 下完全能跑过。哈希板子极短，不易写挂，首选方案B。
- *模板特异化思考*：我现有的哈希板子能提取区间哈希值（`hs.get(l, r)`）。我需要外挂一个二分。二分的是什么？是当前可匹配的最长长度 `mid`。二分的 check 函数就是比较 $S$ 在起点后的 `mid` 长度哈希值，与 $T$ 当前指针后 `mid` 长度哈希值是否相等。

------

### 三、 通用解题方法论（避免脑中空白）

当在比赛中遇到毫无头绪的题时，可以尝试以下这套推导飞轮：

1. **极端化思考（Brute Force & Limit）**：先想出最暴力的解法，哪怕是指数级或 $O(N^3)$ 的。明确这个暴力解法中，哪一步做了**最无用的重复计算**。
2. **寻找“畸形”数据与小常数（Anomaly Detection）**：观察题目中显得格格不入的条件。比如 $K \le 3$、字符集只有 `a-c`、某个权重 $\le 30$。这些“畸形”条件就是突破口，通常对应着状态压缩的维度、分段的次数、或者是某种数据结构的深度。
3. **语义重构（Semantic Reframing）**：把题目的大白话翻译成数据结构语言。
   - “最多错3个” -> “最多切4段求LCP”。
   - “找最接近的值” -> “前驱后继查询”。
   - “连通且权值最大” -> “Kruskal重构树/并查集”。
4. **拼图式组装（Black-box Assembly）**：明确了需要哪些操作（比如本题需要快速区间相等判断），去你的“武器库”里找对应复杂度的模板（Hash/SA/线段树）像黑盒一样拼装进去，最后计算总复杂度。

### 四、 顶尖选手的编码流（思路到代码的无缝转换）

为了避免“写着写着就卡住”，金牌选手通常采用**“自顶向下设计，自底向上实现”**的编码流，全程保持“状态清晰”。你的代码结构已经有了这种雏形，我们来梳理一下标准流程：

**Step 1: 黑盒化依赖（Top-Down）**

在脑中有了解法后，不要一开始就去写二分和哈希的细节。先写主逻辑，假设所有工具函数都已经完美存在。

*内心OS*：我需要枚举起点，然后如果 `match(i)` 成功就 `ans++`。

```cpp
// 脑中先只装这几行
for(int i = 0; i <= n - m; i ++) {
    if(match(i + 1)) ans ++;
}
```

**Step 2: 明确核心状态变量的生命周期**

开始写 `match` 函数。在这个函数里，坚决不要让变量名和状态混淆。你的代码里 `cur` 和 `cnt` 用得非常好。

我们需要明确：

- `cur`：当前在模式串 $T$ 中已经成功匹配的**总长度**。它也是我们下一段 LCP 的起点偏移量。
- `cnt`：剩余容错次数（或者说还能再匹配几段）。初始为 4。

**Step 3: 填补黑盒（Bottom-Up）**

写内部的 LCP 逻辑。这时候只专注二分：

```cpp
// 在 match 内部，心无旁骛地写一个二分 LCP
int l = 0, r = m - cur;
while(l < r) {
    int mid = l + r + 1 >> 1; // 向上取整的二分模板
    if( check(mid) ) l = mid;
    else r = mid - 1;
}
```

**Step 4: 状态推进与退出边界**

LCP 找完后，推进状态 `cur += l`。此时遇到了错误字符，如果还有容错机会 (`cnt > 0`)，强行跳过错误字符 (`cur++`)。最后判断 `cur >= m` 即可。

**总结编码流**：先写出最大的壳（主循环） $\rightarrow$ 定义内部流转的少数几个核心变量（`cur`, `cnt`） $\rightarrow$ 填入数据结构/算法黑盒（二分LCP） $\rightarrow$ 处理边界。不要在写二分的时候去想外层的 $i$ 循环，保持大脑在单层的专注。

### 五、 补充视角：更进阶的思考方向

既然目标是金牌，这道题还有两个非常有价值的延伸视角：

**1. 询问极大情况下的极致优化：迈向 $O(1)$ LCP**

现在的复杂度带一个 $\log$。如果题目数据强化到 $|S|, |T| \le 10^6$，哈希+二分会因为常数和 $\log$ 被卡（或者被精心构造的哈希冲突卡掉）。

*进阶方向*：使用 **后缀数组 (Suffix Array)** 或 **后缀自动机 (SAM)**。将 $S$ 和 $T$ 拼接成 $S + \# + T$，跑出 SA 和 Height 数组，用 ST 表预处理后，任何两段的 LCP 都可以通过 RMQ 在 $O(1)$ 时间内求出。总复杂度降为绝对的 $O(N)$。这是处理高级字符串问题的标准前置技能。

**2. 条件反转：如果允许的错误数很大怎么办？**

如果题目改为：“允许错误 $K$ 个，但字符集非常小（比如只有 DNA 序列的 `A, C, G, T` 4种字符）”。此时分段 LCP 就不适用了（切片太多）。

*进阶方向*：这时候就要用到带有通配符或带权匹配的 **FFT / NTT** 字符串匹配算法了。通过构造多项式，可以在 $O(N \log N)$ 内求出所有起点的匹配度。



## 树上双向哈希倍增跳表

> 维护向下的跳表时，并非维护从当前点向下的信息，而是向下到达当前点的信息

[G-Bingbong的回文路径_牛客小白月赛91](https://ac.nowcoder.com/acm/contest/78807/G)

这道题的完成度非常高。你使用了 $2^{61}-1$ 的大质数取模，并配合随机化哈希 Base，这种写法防碰撞能力极强，是非常成熟的竞赛级哈希模板，说明你在基础模板的积累上已经做得很出色了。

为了达到 ICPC 金牌的水平，仅仅 AC 是不够的，我们需要把这道题“榨干”。以下是我为你整理的深度剖析：

### 一、 题目归类与本质探讨

**算法类型：** 树上倍增（Tree Binary Lifting） + 字符串哈希（String Hashing）

**题目本质：** 树上非交换律信息的路径合并。

这道题的底层逻辑在于处理**不满足交换律**的信息（字符串拼接 $A+B \neq B+A$）。在树上处理路径问题时，如果信息满足交换律（例如求路径最大值、路径和），我们只需要维护一种状态；但如果不满足交换律，我们就必须维护**正反两种状态**（即向上到祖先、向下到子孙）。这道题的核心考察点就是：你是否能熟练地利用倍增思想，将具有方向性的字符串片段正确拼接。

### 二、 思维复盘：无上帝视角的推导过程

假设我们在赛场上第一次看到这道题，思维链应该这样展开：

1. **目标降维：** 题目要求判断树上两点 $u$ 到 $v$ 的路径是否是回文串。首先，树上任意两点路径唯一，且必然经过它们的最近公共祖先（LCA）。因此，路径被天然拆分为两段：$u \to \text{LCA}$ 和 $\text{LCA} \to v$。
2. **条件转换（核心破局）：** 怎么判断回文？常规想法是找中点看两边，但在树上找中点成本极高（需要算距离求 $k$ 级祖先）。回归回文串的最基础数学定义：一个字符串 $S$ 是回文串，当且仅当它**等于它的逆序串** $S^R$。
   - 映射到树上，即：如果 `Path(u -> v)` 的构成的字符串，等于 `Path(v -> u)` 构成的字符串，它就是回文的。
   - 结论：我们只需要一种能够快速比较两个字符串是否相等的数据结构 $\to$ **字符串哈希**。
3. **算法定位与改造：** 我们需要快速求出树上任意路径的哈希值。因为树是静态的，且有多次询问，自然想到 $O(n \log n)$ 预处理、$O(\log n)$ 查询的**树上倍增**。
   - **常规倍增模板：** `f[y][i] = f[f[y][i-1]][i-1]`。
   - **特异化修改（自我问答）：**
     - *问：拼接两个哈希值需要什么条件？* 答：需要知道后半段的长度，以决定前半段乘上 Base 的多少次方。
     - *问：倍增跳跃时，方向一致吗？* 答：不一致。从 $u \to \text{LCA}$ 是向根节点走，从 $\text{LCA} \to v$ 是背离根节点走。
     - *问：怎么维护这两种走法？* 答：开两个倍增数组。`stup[y][i]` 维护向上跳 $2^i$ 步的哈希（前半段是下面，后半段是上面）；`stdown[y][i]` 维护从上面向下走 $2^i$ 步的哈希（前半段是上面，后半段是下面）。

### 三、 你的初次思路差在哪？

你的初次想法是：“得到树链长度 $\to$ 找中点断开 $\to$ 判断前缀和后缀是否相等 $\to$ 不会维护从下往上的哈希，卡住”。

这个思路的根源在于**被线性数组的思维定式禁锢了**。在数组里，找中点是 $O(1)$ 的本能反应。但在树上，寻找“中点”需要求 LCA、计算深度差、再分类讨论从 $u$ 向上跳还是从 $v$ 向上跳，逻辑极其冗长且容易写出 Bug。

**正确的思维拐点在于“整体大于局部”：**

不要去分割字符串，而是利用哈希 $O(1)$ 比较的能力，直接比较**整条正向路径**与**整条反向路径**。至于你提到的“不会维护从下往上的哈希”，其实一旦你想通了“倍增可以向上跳，只是拼接时的先后顺序不同”，这个问题就迎刃而解了。你现在的 AC 代码中 `stup` 和 `stdown` 的转移方程，正是完美解决了这个所谓“从下往上”的难题。

### 四、 通用解题方法论

为了避免遇到新题时大脑空白，你可以建立以下“剥洋葱”式的思考流：

1. **第一层（操作抽象）：** 题目要求我做几次操作？是静态查询还是动态修改？（决定了是用倍增、树剖还是 LCT）。
2. **第二层（信息性质）：** 题目要求维护的数据（和、最值、连通性、字符串）满足什么代数性质？
   - 满足交换律？$\to$ 随便拆分合并。
   - 不满足交换律？$\to$ 必须维护两个方向的状态（经典如线段树维护最大子段和、倍增维护哈希）。
3. **第三层（重新定义）：** 当常规思路（如找中点）遇到巨大实现阻力时，立刻后退一步，回想该概念的最基本定义。回文不仅是“中心对称”，更是“正反读完全一致”。很多难题的本质就是考察对定义的等价转换。

### 五、 顶尖选手的编码流设计

在赛场上，写倍增往往容易把转移细节写错导致疯狂 Debug。高手的编码流通常是这样的：

1. **基建先行（肌肉记忆）：** 先把取模加法、减法、乘法以及哈希的 `init_hash()` 写好。确保所有底层工具函数绝对正确（你的 `add`, `sub`, `mul` 就做得很好）。
2. **核心转移方程的纸笔推演（防卡壳关键）：** 在敲击键盘写 BFS/DFS 之前，先在草稿纸上画三个点 $A \to B \to C$。
   - 明确写下 `stup[y][i]` 的组合：先经历了 $y \to f[y][i-1]$，再经历了 $f[y][i-1] \to f[y][i]$。对应哈希组合是：底下的哈希 $\times base^{len} +$ 上面的哈希。
   - 明确写下 `stdown[y][i]` 的组合：先经历了 $f[y][i] \to f[y][i-1]$，再经历了 $f[y][i-1] \to y$。对应哈希组合是：上面的哈希 $\times base^{len} +$ 底下的哈希。
   - 看着草稿纸上的公式，一鼓作气写完树的遍历和倍增预处理。
3. **解耦查询逻辑：** 不要把复杂的路径哈希计算和 LCA 混在一起。写一个干净的 LCA 函数只负责找点。再写一个 `get(x, y, P)` 函数专门负责将 $x$ 沿倍增跳到 $P$ 的信息合并。
4. **注意长度的维护：** 倍增合并信息时，最容易错的就是哈希的权重（要乘上 $P[len]$）。在写 `get` 函数内部向上跳的过程中，一定要显式地维护当前已经跳了多长（如你代码中的 `res += 1 << i`），这是保持逻辑清晰的不二法门。
