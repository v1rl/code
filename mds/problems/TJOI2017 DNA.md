### 洛谷 P3763 [TJOI2017] DNA - 三种解法总结

本题的核心痛点在于“允许容错（最多 3 次失配）”。针对这一特性，主要衍生出了两种大方向：**“跳过失配点（求 LCP / LCS）”** 和 **“字符集独立的卷积匹配”**。

#### 1. 解法一：字符串 Hash + 二分查找 LCP

- **核心思路**：

  枚举主串 $S$ 中的匹配起点 $i$。在匹配过程中，我们需要快速跳过完全相同的连续字符。通过预处理字符串的哈希值，我们可以使用二分查找在 $O(\log N)$ 的时间内求出从当前位置开始的**最长公共前缀 (LCP)**。如果遇到不匹配的字符，就强行跳过（容错次数减 1），最多跳跃 4 次。

- **时间复杂度**：$O(N \log |T|)$

- **优点**：思维难度极低，代码实现相对直观，常数较小，在大多数比赛中是性价比最高的骗分甚至 AC 利器。

- **缺点**：极其依赖 Hash 函数的质量。本题数据可能会卡单哈希，你使用了 `mt19937_64` 动态生成 base 和模 $2^{61}-1$ 的大质数（利用 `u128` 防溢出）是非常稳妥的防卡策略。

#### 2. 解法二：多项式乘法 (NTT) 字符集卷积

- **核心思路**：

  将问题转化为“求完全匹配的字符个数”。由于字符集很小（只有 `A, T, C, G` 四种），我们可以对每种字符单独处理。对于某种字符，将 $S$ 和 $T$ 中该字符出现的位置设为 1，其余设为 0。将 $S$ 反转后，两者进行多项式乘法（NTT），结果多项式的第 $k$ 项系数，就代表了 $T$ 相对于 $S$ 偏移一定位置时，该字符的匹配数量。将 4 种字符的匹配数相加，若总匹配数 $\ge |T| - 3$，则说明该位置合法。

- **时间复杂度**：$O(|\Sigma| N \log N)$，其中 $|\Sigma| = 4$。

- **优点**：**绝对正确，没有 Hash 碰撞的风险**。这种做法是带有通配符或定长模糊匹配问题的最标准、最普适的数学解法。

- **缺点**：常数巨大（需要跑 4 次完整的 NTT），内存开销也比较大。如果题目限制了极端的时空，可能会被卡常。

#### 3. 解法三：后缀自动机 (SAM) + 树上 RMQ 求 LCS

- **核心思路**：

  总体逻辑与“解法一”类似，依然是“匹配 + 跳过失配点”。但为了把求 LCP 的时间复杂度从 $O(\log N)$ 降到 $O(1)$，使用了 SAM。

  将 $T$ 和 $S$ 拼接（中间用 `#` 隔开）构建 Suffix Automaton。在 SAM 的 Parent Tree（Link 树）上，两个节点代表的子串的**最长公共后缀 (LCS)** 的长度，等于它们在 Parent Tree 上的 LCA 的 `len`。通过 DFS 序 + ST 表预处理 RMQ 问题，可以实现 $O(1)$ 查询 LCS 长度，从而极其高效地完成跳过操作。

- **时间复杂度**：预处理 $O(N \log N)$，单次匹配查询 $O(1)$，总复杂度 $O(N \log N)$。

- **优点**：匹配阶段达到了理论下界 $O(1)$，非常优雅。深刻利用了 SAM Parent Tree 的性质，是字符串高级数据结构的巅峰应用之一。

- **缺点**：杀鸡用牛刀。代码量最大，SAM 的节点数是 $2N$，ST 表的预处理不仅常数大且耗费内存，在实际运行速度上未必能完全碾压 Hash+二分。

------

### 总结论

- **从赛场实用性来看**：**Hash + 二分** 绝对是首选，代码短、速度快，只要 Hash 选得好（双哈希或随机大质数），性价比无敌。
- **从算法严谨性来看**：**NTT 卷积匹配** 提供了不依赖概率的确定性做法，且这种思想可以平滑过渡到“带通配符的字符串匹配”问题中。
- **从数据结构上限来看**：**SAM + RMQ** 展现了极强的字符串处理能力，当容错次数变多或者需要进行大量 $O(1)$ LCP 询问时，SAM/SA 无疑是必须掌握的核心武器。
