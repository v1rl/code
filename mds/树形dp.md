## 标准树上背包

[P2015 二叉苹果树 - 洛谷](https://www.luogu.com.cn/problem/P2015)

这道题（P2015 二叉苹果树）虽然数据范围较小（$N \le 100$），但它是**树上背包（Tree Knapsack）**最经典的入门题。

以下是针对你的四个维度的深度复盘：

> 使用siz进行上下界优化时，输出最终答案需要注意m有没有可能大于siz[1]，如果有可能则应该输出`f[1][min(n, siz[1])]`

### 1. 题目归类与本质：这是什么？

- **一级归类**：动态规划（DP）

- **二级归类**：树形 DP（Tree DP）

- **核心模型**：**树上背包问题（有依赖的背包问题）**

- **本质描述**：

  这道题的本质是在**树的拓扑结构约束**下进行**资源分配**。

  - **物品**：子树（或者说连向子节点的边及其背后的子树）。
  - **体积**：保留的树枝数量（$M$）。
  - **价值**：苹果的数量。
  - **依赖性**：如果你想获得子节点 $v$ 子树内的收益，你必须先花费 1 个单位体积（保留边 $u \to v$）来“解锁”进入该子树的资格。

**以后遇到这类题的直觉：**

看到“树结构”、“选 $k$ 个点/边”、“最大化权值之和”这三个关键词组合时，90% 的概率是**树上背包**。

------

### 2. 从零开始的思维复盘：怎么推导出解法？

假设我们没有上帝视角，怎么一步步逼近这个状态转移方程？

**阶段一：捕获关键条件与初步尝试**

- **条件**：一棵树，边有权重，只能留 $Q$ 条边，求最大权值。
- **直觉 1（贪心）**：是不是只选权值最大的边就行？
  - *反驳*：不行。如果一条边权值很大（比如 100），但它接在一条权值很小（比如 1）的边后面，要选那条大边必须选前面的小边。**树的连通性**是硬约束。
- **直觉 2（搜索）**：爆搜所有组合？$C(N, Q)$ 太大，会超时。
- **结论**：具有最优子结构（子树的最优解能推导出全树最优解）和重叠子问题，确定使用 **DP**。

**阶段二：定义状态（目标条件的转换）**

- 我们需要什么信息才能做决策？
  - 我在哪个节点（位置信息 $u$）。
  - 我还剩多少配额可以用（资源信息 $j$）。
- **状态定义**：$dp[u][j]$ 表示以 $u$ 为根的子树中，保留 $j$ 条边能获得的最大苹果数。
  - *为什么是保留 $j$ 条边而不是节点？* 题目限制的是边（树枝），且边依附于节点关系存在。

**阶段三：寻找转移（分组背包的映射）**

- 这是最关键的一步。站在节点 $u$ 上，我有若干个子节点 $v_1, v_2, \dots$。
- 我有 $j$ 个树枝的额度，我要把这些额度**分**给这几个孩子。这像什么？
  - 把 $u$ 看作一个**背包**，容量为 $j$。
  - 每个子节点 $v$ 及其子树看作一组**物品**。
  - 但这组物品很特殊：我可以给 $v$ 分配 $0$ 个树枝，也可以分配 $1$ 个，...，或者 $k$ 个。
  - 如果我们给子树 $v$ 分配 $k$ 个树枝，实际消耗是 $k+1$（因为要算上 $u \to v$ 这条连接边），获得的价值是 $dp[v][k] + w_{u \to v}$。
- **本质转化**：这就是一个**分组背包问题**！
  - 每个子节点是一个“物品组”。
  - 在这个组里，我只能决策选一种策略（给它分配具体多少容量）。

**阶段四：方程推导**

先将siz[y]合并：$$dp[x][j] = \max(dp[x][j], \quad dp[x][j - k - 1] + dp[y][k] + w_{u \to v})$$

或者先执行后合并：$$dp[x][j + k] = \max(dp[x][j + k], \quad dp[x][j - 1] + dp[y][k] + w_{u \to v})$$

- 这里 $dp[u][j]$ 的更新类似于背包的滚动数组，指的是“考虑前 $i$ 个儿子时的最优解”。

### 3. 通用解题方法论：遇到陌生树形DP如何下手？

当你面对一道陌生的树形题目（尤其是涉及计数或最值）时，请按以下**“树形DP三部曲”**进行剖析：

#### 第一步：确定“子树信息”

问自己：**“如果我已经算出了所有子节点 $v$ 的答案，我还需要知道关于 $v$ 的什么额外信息，才能算出当前节点 $u$ 的答案？”**

- 在本题中，只知道子树的最大值是不够的，还必须知道“子树用了多少条边”才能合并。所以状态里必须带上“体积”这一维。

#### 第二步：设计“合并逻辑”（Merge Policy）

树形 DP 的核心就是 **DFS 后序遍历**（先处理子节点，再回溯处理父节点）。在代码实现上，通常有两种通用的合并模板：

1. **普通选取类**（如最大独立集、没有上司的舞会）：
   - 当前节点选/不选，直接由子节点选/不选转移而来。
   - $O(N)$。
2. **背包资源类**（如本题）：
   - 涉及资源分配。需要两层循环：外层枚举当前子树总体积，内层枚举给该子节点分配的体积。
   - $O(N^3)$ 或 $O(N^2)$。

#### 第三步：边界与细节检查

- **初始化**：叶子节点的 DP 值是多少？非法状态（如体积为负）是否设为了 $-\infty$？
- **边的处理**：边权是算在节点上，还是算在转移代价里？（本题算在转移代价里）。

### 5. 补充：为什么复杂度是 $O(N^2)$？

你可能会疑惑，两层循环嵌套在 DFS 里，看起来像 $O(N^3)$？

这是一个经典的面试/ICPC 考点。

**证明：**

在 `f[u][j+k] = ...` 的合并过程中，每一对节点 $(a, b)$ 只会在它们**最近公共祖先 (LCA)** 处被合并一次。

- 当我们在 $u$ 节点合并 $u$ 的已有连通块和 $v$ 的子树时，实际上是在尝试匹配“$u$ 左边子树里的节点”和“$v$ 子树里的节点”。
- 任意两个节点 $a, b$ 在整个算法过程中，只会在 $LCA(a, b)$ 处作为“左右两部分”被计算一次。
- 节点对的总数是 $N \times N$。
- 所以带有 `sz` 优化的树形背包复杂度严格是 $O(N^2)$。



## DFS序背包

### 1. 题目归类与本质：这是什么？

- **一级归类**：动态规划（DP） / 背包问题

- **二级归类**：**有依赖的背包问题 (Dependency Knapsack)**

- **特例归类**：**分组背包 (Group Knapsack)**

  - *注：本题由于限制了主件最多 2 个附件，通常被视为分组背包处理；但如果附件数量不限，就是标准的树形 DP。*

- **本质描述**：

**金牌视角直觉：**

- 看到“购买 $i$ 必须购买 $p_i$” $\rightarrow$ **森林（多棵树）结构**（$p_i=0$ 为虚拟根节点）。
- 看到“主件 + 附件”组合 $\rightarrow$ **决策空间的枚举**。

### 2. 从零开始的思维复盘：推导与降维

假设我们面对这道题，如何一步步推导解法？

**阶段一：识别依赖与冲突**

- 这是一个背包问题，但物品之间不独立。
- $A$ 依赖 $B$。意味着我们不能单选 $A$。
- **常规思路**：把 $B$ 和它的附件们看作一个整体。

**阶段二：维度的权衡（关键分歧点）**

这里有两种处理依赖关系的流派，决定了代码的走向：

- **流派 A：泛化处理（你的做法 - 树形DP）**
  - **思维**：不管多少附件，反正这就是一棵树。我想在以 $u$ 为根的子树里花 $j$ 元钱，能得到的最大价值。
  - **推导**：进入子树 $v$ 必须先选 $u$。
  - **优点**：通用性强，哪怕一个主件有 100 个附件也能做。
  - **缺点**：状态转移稍微复杂，常数稍大。
- **流派 B：利用题目特殊性质（通常解法 - 分组背包）**
  - **捕获关键条件**：“每个主件最多有 2 个附件”。
  - **转换**：这个条件暗示我们，一个主件及其附件的**组合状态非常少**。
  - **枚举状态**：对于主件 $u$ 和附件 $v_1, v_2$，只有以下 5 种互斥的选择策略：
    1. 不买 $u$（也就不能买 $v_1, v_2$）。
    2. 只买 $u$。
    3. 买 $u + v_1$。
    4. 买 $u + v_2$。
    5. 买 $u + v_1 + v_2$。
  - **模型映射**：这完美的对应了**分组背包问题**——有 $N$ 组物品（每个主件系列是一组），每组里只能选 1 个“复合物品”（上述 5 种策略之一）。
  - **目标条件**：将依赖关系“压扁”成互斥的物品组。

### 3. 通用解题方法论：处理“有依赖的背包”

遇到“物品之间有先决条件（Pre-requisite）”的背包问题，按以下流程思考：

#### 方法一：状态压缩/分组（适用于“依赖层级浅”或“依赖个数少”）

- **场景**：主件最多带 $k$ 个附件，或者依赖关系仅限于 $A \to B$ 两层。
- **操作**：将“主件 + 所有可能的附件子集”打包成多个互斥的虚拟物品。
- **复杂度**：$O(N \cdot V \cdot 2^k)$。当 $k$ 小时极快。

#### 方法二：DFS 序背包（你的解法，通用且高效）

- **场景**：一般树形结构，依赖关系复杂。
- **操作**：
  1. 先将树转化成 **DFS 序（先序遍历）**。
  2. 在递归进入子节点前，将父节点的背包状态在强制选子节点的情况下拷贝给子节点，递归字节点，子节点算完后，再与父节点原状态取 max（决策：要么不进y子树(原f[x])，要么进y子树(f[y])）
- **复杂度**：$O(N \cdot V)$。这是一个非常优秀的复杂度（比传统树形DP合并的 $O(V^2)$ 要好）。

#### 方法三：传统树形 DP 合并（适用于 $V$ 极小）

- **操作**：$dp[u][j] = \max(dp[u][j], dp[u][j-k] + dp[v][k])$。
- **复杂度**：暴力$O(N \cdot V^2)$，siz优化$O(V^2)$。在本题 $V=32000$ 的情况下会 TLE，但在 P2015 那种 $V=100$ 的题里是正解。

### 4. 顶尖选手的编码流（以“分组背包”为例）

既然你已经掌握了通用的树形 DP 写法，我这里演示一下如果这是场比赛，为了最快 AC 且代码量最小，金牌选手通常会怎么写这道题（分组背包思路）。

**Step 1: 预处理与压缩 (Compression)**

> *思维：* "钱数 $N$ 是 32000，都是 10 的倍数。为了优化常数和空间，我读入时直接把钱除以 10。$N$ 变成了 3200。"

**Step 2: 数据结构存储 (Grouping)**

> *思维：* "不要建图了，太麻烦。直接开个数组存主件，再开个 `vector` 存挂在它下面的附件。主件 $i$ 的附件列表 `servant[i]`。"

**Step 3: 核心 DP (The Logic)**

> *思维：* "这是分组背包。外层枚举主件（组），中层枚举体积（从大到小，0/1背包套路），内层枚举该主件的策略（0/1/2/1+2）。"

C++

```
// 顶尖选手的极简流写法
void solve_group_knapsack() {
    int n, m;
    cin >> n >> m;
    n /= 10; // 优化：价格缩小10倍
    
    // items[i] = {v, p, q}
    // servants[main_id] 存附件的索引
    vector<array<int, 3>> items(m + 1);
    vector<vector<int>> servants(m + 1);
    
    for(int i = 1; i <= m; i++) {
        int v, p, q;
        cin >> v >> p >> q;
        v /= 10;
        items[i] = {v, v * p, q};
        if (q != 0) servants[q].push_back(i);
    }

    vector<int> f(n + 1, 0);

    for (int i = 1; i <= m; i++) {
        if (items[i][2] != 0) continue; // 不是主件跳过

        // 分组背包：每组只能选一种策略，所以体积从大到小枚举
        for (int j = n; j >= items[i][0]; j--) {
            // 策略0：只买主件
            int main_v = items[i][0];
            int main_w = items[i][1];
            f[j] = max(f[j], f[j - main_v] + main_w);

            // 策略1：主件 + 附件1
            if (servants[i].size() >= 1) {
                int s1 = servants[i][0];
                int v1 = items[s1][0] + main_v;
                int w1 = items[s1][1] + main_w;
                if (j >= v1) f[j] = max(f[j], f[j - v1] + w1);
            }

            // 策略2：主件 + 附件2
            if (servants[i].size() >= 2) {
                int s2 = servants[i][1];
                int v2 = items[s2][0] + main_v;
                int w2 = items[s2][1] + main_w;
                if (j >= v2) f[j] = max(f[j], f[j - v2] + w2);
            }

            // 策略3：主件 + 附件1 + 附件2
            if (servants[i].size() >= 2) {
                int s1 = servants[i][0], s2 = servants[i][1];
                int v3 = items[s1][0] + items[s2][0] + main_v;
                int w3 = items[s1][1] + items[s2][1] + main_w;
                if (j >= v3) f[j] = max(f[j], f[j - v3] + w3);
            }
        }
    }
    cout << f[n] * 10 << endl; // 结果不用乘10，因为value没除，是cost除以10了。如果不小心除错了，这里注意一下。
    // 修正：实际上题目里 w[i] = v[i] * p[i]，如果 v 除以了 10，w 最好不要变，或者 w 也除以 10 最后乘回来。
    // 建议：只把体积维度的 32000 变成 3200，价值维度不要动。
}
```

### 5. 对你代码的深度分析（Expert Review）

你的代码使用了一种很巧妙的 **"副本-递归-合并"** 策略，

传统树形dp没有自顶向下的过程，合并状态时枚举代价（自底向上选多少）`f[在子树x中][代价最多为j]`的最大收益
这里先继承并强制选（自顶向下），合并时直接考虑是否进入子树（自底向上选不选）`f[在dfs序中遍历到x时][代价最多为j]`的最大收益

```cpp
// 你的核心逻辑
f[y][j] = f[x][j - v[y]] + w[y]; // 1. 强制买y，继承x的状态
self(self, y);                   // 2. 在y的子树里继续买
f[x][j] = max(f[x][j], f[y][j]); // 3. 决策：要么不进y子树(原f[x])，要么进y子树(f[y])
```

这种写法其实非常高明，它巧妙地避开了 $O(M^2)$ 的合并，做到了线性（针对背包容量）的复杂度。

------

### 1. 题目归类与本质：这是什么？

- **一级归类**：动态规划 (Dynamic Programming)

- **二级归类**：树形 DP (Tree DP)

- **本质描述**：

  这是一个**连通块提取**问题。

  题目本质是在一棵树中找到一个**大小为 $P$ 的连通子图**（Connected Subgraph），使得该子图与剩余部分断开的边数最少。

  *注意：虽然题目说是“子树”，但结合题意，其实是指以任意节点为根的连通块。*

**以后遇到这类题的直觉：**

- 关键词：“保留 $P$ 个节点”、“最少删去多少边”、“连通性”。
- 一旦涉及“在树上选 $K$ 个连通的点”，99% 是树形背包模型（$dp[u][k]$ 表示以 $u$ 为根选 $k$ 个）。

------

### 2. 从零开始的思维复盘：怎么推导出解法？

假设我们没有上帝视角，如何一步步逼近你现在的代码逻辑？

**阶段一：捕获关键条件与初步转化**

- **条件**：要一个 $P$ 个节点的子树（连通块）。

- **目标**：最少的“被剪断”的边。

- **难点**：直接数“剪断的边”很难，因为剪断的边分布在连通块的边缘。

- **思维转换（补集思想/度数转换）**：

  与其想“剪断了谁”，不如想“保留了谁”。

  对于一个选定的点集 $S$，我们要计算它与 $S$ 以外的点的连边数量。

  有一个重要的图论性质：

  $$\text{Cut Edges} = \sum_{v \in S} \text{degree}[v] - 2 \times (\text{Edges inside } S)$$

  这个公式是对的，但写进 DP 里有点麻烦（要同时维护点数、度数和内部边数）。

**阶段二：寻找更优的状态定义（你的神来之笔）**

- 能不能把“割边”的代价局部化？
- **初始化视角**：假设我只选节点 $u$ 一个点。那么与 $u$ 相连的所有边都要断开。此时代价是多少？是 $u$ 的度数 `d[u]`。
- **合并视角**：现在的状态是 $u$ 单独一个点（代价 `d[u]`）。如果我要把 $u$ 的儿子 $v$ 并入 $u$ 的连通块，会发生什么？
  1. $u$ 不用切断和 $v$ 的边了（代价 -1）。
  2. $v$ 也不用切断和 $u$ 的边了（代价 -1）。
  3. $v$ 原本计划切断的其他边（即 $v$ 的度数减去与 $u$ 连接的那条）依然要切断。
- **结论**：合并一个子连通块的代价变化是 **-2**（少切了一条 $u-v$ 的边，这条边在 $u$ 算了一次，在 $v$ 算了一次）。

**阶段三：确定 DP 状态**

- 我们需要知道当前根是谁（$u$），以及当前连通块多大（$j$）。

- $f[u][j]$：以 $u$ 为根，保留 $j$ 个节点的连通块，需要切断的边数。

- **状态转移方程**（也就是树上背包的标准形式）：

  $$f[u][j+k] = \min(f[u][j+k], \quad f[u][j] + f[v][k] - 2)$$

  - $f[u][j]$：$u$ 已经选了 $j$ 个。
  - $f[v][k]$：儿子 $v$ 选了 $k$ 个。
  - $-2$：因为 $(u, v)$ 这条边变成了内部边，不用切了。

**阶段四：边界处理**

- 题目要求的是“得到一个 $P$ 个节点的子树”。
- 这个子树的根可以是树上的任意节点 $i$。
- 如果选出的连通块根是 $i$，且 $i$ 不是整棵树的 Root，那么 $f[i][P]$ 里是否包含了切断 $i$ 与 $parent(i)$ 的代价？
- **Check**：初始 `f[i][1] = d[i]`。`d[i]` 包含了 $i$ 到 $parent(i)$ 的边。在 DFS 过程中，如果不选 $parent(i)$（即 $i$ 是作为最终连通块的根），那么 $parent(i)$ 不会通过转移方程把 $-2$ 贡献给 $i$。所以 `d[i]` 里包含的“切断父节点边”的代价被保留了下来。**逻辑闭环，完全正确。**

------

### 3. 通用解题方法论：树上背包问题

以后遇到**“树上选 $K$ 个点/边”**或者**“树上资源分配”**的问题，直接套用以下四步法：

**Step 1: 定义状态**

通常是 `dp[u][k]`，表示以 $u$ 为根的子树中，选取了 $k$ 个（节点/体积/成本）的（最大价值/最小代价）。

**Step 2: 确定初始化 (Base Case)**

- 叶子节点或仅选根节点时的值。
- *本题特色*：`dp[u][1] = degree[u]`。
- *常见情况*：`dp[u][0] = 0` 或 `dp[u][1] = value[u]`。

**Step 3: 编写循环框架 (The Loop)**

这是最容易写错的地方，务必背诵以下结构：

```cpp
for(auto y : adj[x]) {
    self(self, y);
    for(int j = min(m, siz[x]); j >= 1; j --) {
        for(int k = min(m - j, siz[y]); k >= 1; k --) {
            f[x][j + k] = min(f[x][j + k], f[y][k] + f[x][j] - 2);
        }
    }
    siz[x] += siz[y];
}
```

**Step 4: 答案统计**

- 如果是“必须包含根节点”，答案通常是 `dp[root][P]`。
- 如果是“任意连通块”（如本题），答案通常是 `min/max(dp[i][P])` for all $i$。

------

### 4. 顶尖选手的编码流 (Coding Flow)

金牌选手在写这道题时，为了保证 **Bug-free** 和 **Speed**，思维流是这样的：

1. **I/O 与 图存储**：

   > "无向图，$N=150$。直接 `vector<int> adj[N]`。要用到度数，读入时顺便统计 `d[x]++, d[y]++`。"

2. **DP 数组初始化**：

   > "求最小值，初始化 `inf`。`f[u][1]` 显式初始化为 `d[u]`。这一步不能在 DFS 里漏掉。"

3. **DFS 框架 (核心)**：

   > "写 DFS。先递归 `dfs(v)`。
   >
   > 接着是背包合并。
   >
   > **关键点**：我要枚举两个体积，一个是 $u$ 当前的体积 $j$，一个是 $v$ 的体积 $k$。
   >
   > **防坑**：$j$ 必须从大到小循环！$j$ 从 `sz[u]` 到 1。$k$ 从 1 到 `sz[v]`。
   >
   > 转移方程：`f[u][j+k] = min(..., f[u][j] + f[v][k] - 2)`。"

4. **复杂度控制 (Size Optimization)**：

   > "虽然 $N=150$ 随便跑，但养成好习惯，循环上界要用 `sz[u]` 和 `sz[v]` 控制。这样复杂度是 $O(N^2)$ 而不是 $O(N^3)$。"

   - *注：你现在的代码里虽然写了 `siz`，但如果遇到 $N=3000$，这种写法就是生死的区别。*

5. **全局答案 Check**：

   > "题目没说根是 1，遍历 $1 \dots N$，取 `f[i][m]` 的最小值。"

### 5. 为什么本题不能DFS序背包

1. **父->子拷贝**约束是“选子必须选父”，所以**子节点的可行状态本来就依赖父节点是否被选**。

因此可以：

- 先从父节点状态出发初始化子节点（“已经选了 y”）
- 再在 y 子树内部扩展
- 回来合并

这就是你之前那种“父状态灌到子状态”的来源：**子问题必须带着父亲的选择前提**。

2. P1272（删边分离出大小为 P 的连通块）

你这题的经典状态（你代码也是这个）是：

- `f[x][j]`：在以 x 为根的子树里，最终保留下来一个连通块（包含 x）且大小为 j 时，需要删的最少边数。

这个状态的关键点
` f[y][k] `完全由 y 的子树内部结构决定，和父节点当前“打算保留多少点”无关。

父亲 x 想要形成大小 j 的连通块，本质是把“名额” `j-1`在各个孩子子树之间分配：

- 从某个孩子 y 里拿 k 个点，就要用到 `f[y][k]`

这里 **k 是在合并时才枚举决定的**，不是递归进入 y 之前就能确定的。
**P1272 的“决策”发生在父节点合并阶段（分配 k），而不是发生在进入子节点之前。**
所以没法像 P1064 那样，先把父节点信息直接拷贝到子节点，让子节点“带着父亲的决策”去算。



# 维度下标与属性价值的选择

[P1273 有线电视网 - 洛谷](https://www.luogu.com.cn/problem/P1273)

------

## 1）题目归类：它到底是哪一类 DP

这题的核心是：在一棵树上选择尽可能多的叶子用户，让**总收益 − 总传输成本 ≥ 0**，并且为了给用户送信号，必须保留从根到这些用户的路径边（自然形成连通子树）。题面明确：结构是树、叶子为用户、边有传输费用、每个用户愿意付费，目标是不亏本下用户数最多。([luogu.com.cn](https://www.luogu.com.cn/problem/P1273))

可以给它贴这些“本质标签”：

1. **树形 DP（Tree DP） + 选择连通子树（rooted connected subgraph）**
   选择的用户对应一棵“以根为源、覆盖这些叶子路径”的连通子树。
2. **树上分组背包 / 树形背包（Tree Knapsack）**
   在每个内部节点，要把“从各个孩子子树里选多少用户”做分配（背包卷积/合并）。
3. **目标是“最大数量，约束是“利润非负”** → **经典的“对偶化/换维度”DP**
   你最终 AC 的关键就是：**不要用“钱”做背包维度**，而是用“用户数”做维度，让状态变成“选 k 个用户时最大利润”。

**核心考点**： **“价值”与“体积”的互换（Knapsack Duality）**。 在背包问题中，我们通常计算“给定容量 $W$ 下的最大价值 $V$”。但当 $W$ 很大而 $V$ 很小时，我们需要反过来计算“获得价值 $V$ 所需的最小容量 $W$”。 本题中，不亏本是约束，用户数是目标；但由于钱数范围不可控，我们将“用户数”作为背包容量，“钱数”作为价值。

------

## 2）从零推导复盘：没有题解视角时怎么一步步想到“人数维度 DP”

### Step A：把题面目标改写成最干净的数学形式

选一批用户集合 $$S$$（叶子集合），为了给他们送信号必须选择连接到根的边集合（就是这些叶子到根路径的并）。

- 收入：$$\sum_{u\in S} pay[u]$$（用户愿意付的钱）
- 成本：$$\sum_{e \in E(S)} cost[e]$$（被使用的边的传输费用）
- 约束：$$\sum pay - \sum cost \ge 0$$（不亏本）([luogu.com.cn](https://www.luogu.com.cn/problem/P1273))
- 目标：最大化 $$|S|$$

这一步很重要：**你要立刻看到“这是利润约束下最大化数量”**。

------

### Step B：识别“连通性”其实是树上的天然子结构

在树上，“为了让用户收到信号”意味着**必须包含根到该叶子的路径**。
所以最终选择结构一定是：**以根为根的某棵连通子树**。

于是自然想到：树上自底向上的 DP，子树可以合并。

------

### Step C：关键选择——背包维度选什么？

这题最容易走岔路的点就在这里。

- 你可以把“钱/利润”当维度，状态是 `dp[u][money] = max users`
  ✅ 逻辑上可行
  ❌ 但 money 的范围是“总成本/总收入”，虽然单次费用 ≤ 10，节点最多 3000，但总量级依然可到 $$O(30000)$$（边数约 3000，每条 ≤10）。题面也写了单次传输成本和用户支付均 ≤ 10。([luogu.com.cn](https://www.luogu.com.cn/problem/P1273))
  → **二维数组就是 $$3000 \times 30000$$ 级别，int 直接爆内存**（这就是你 MLE 的根源）。
- 更聪明的方式：把“人数”当维度
  定义：
  $$dp[u][k] = \text{在 }u\text{ 子树内选 }k\text{ 个用户时，最大利润}$$
  这样 $$k \le M \le 3000$$，维度天然小很多。([luogu.com.cn](https://www.luogu.com.cn/problem/P1273))
  最终答案就是最大的 $$k$$ 使得 $$dp[1][k] \ge 0$$。

**思维转换**：既然无法把“钱”当数组下标，那就把“人”当数组下标。

**重新定义**：$dp[u][j]$ 表示以 $u$ 为根的子树中，**恰好**选了 $j$ 个用户，所能获得的**最大净收益**。

**目标检查**：算出 $dp[1][0 \dots M]$ 后，我只要倒着遍历 $j$ 从 $M$ 到 $0$，找到第一个 $dp[1][j] \ge 0$ 的 $j$ 就是答案。

------

## 3）你初版思路 vs 正确思路：差在哪、下次怎么纠偏

你初版想法是：

> “题目要不亏本下最多用户数 → 树形背包 → 费用做属性，维护最大用户数 → 从不亏本费用中取最大用户”

这个思路**方向其实没错**：树形背包 + 利润约束。
真正的差别在于：

### 差别 1：你把“约束量（钱）”当成了 DP 维度

这会导致 `dp[u][money]` 的 money 量级很大（到 3e4），每个节点都开一条，内存直接炸（你 MLE 就是这里）。这是树形背包最常见的“第一坑”。

**下次遇到类似题目的修正策略：**

1. **先看数据范围**：哪一个变量小（$\le 5000$），哪一个变量大（$\ge 10^5$ 或浮点数）。
2. **小即下标**：把那个小的变量强行塞进 DP 状态的维度里。
3. **大即价值**：把那个大的、或者求最值的变量放在 DP 的存储值里。

------

## 4）通用解题方法论：陌生题不至于脑中一片空白的套路

给你一套我建议你刷题时“强制执行”的 DP 诊断流程（尤其适合 ICPC）：

1. **先写出：目标函数 + 约束**（用一个式子落地）
   这题就是 $$\max |S| \text{ s.t. } \sum pay - \sum cost \ge 0$$。
2. **判断结构：能否分解为子结构？**
   树/图/区间/集合？
   这题是树 → 明确“子树合并”。
3. **决定 DP 维度：优先选“范围小”的那个**
   - 预算维度大（可到 3e4）
   - 用户数维度小（≤3000）
     → 选人数维度。
4. **想清楚每次合并时，是否需要“卷积”**
   多孩子分配名额 → 典型卷积/分组背包。
5. **最后把“答案形式”固定成判定**
   求最大 $$k$$ 使 $$dp[root][k] \ge 0$$。

你会发现：这 5 步走完，基本已经在写代码了。
